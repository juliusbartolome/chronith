# Chronith v0.1 — Architecture Design Document

> Status: Approved  
> Date: 2026-03-01  
> Author: Principal Architect

---

## 1. Overview

Chronith is a multi-tenant booking engine supporting two booking modalities: time-slot bookings (fixed duration with optional buffers) and calendar-day bookings (whole-day blocks). Tenants are resolved from JWT. The system is built for performance with strict response time targets and full TDD discipline.

## 3. Domain Model

### 3.1 BookingType — Two separate domain classes, single EF entity

Both types share a common abstract base. EF maps to a single `BookingTypes` table with a `Kind` discriminator. The EF entity (`BookingTypeEntity`) is internal to Infrastructure and never exposed outside it.

```
abstract BookingType
  Id: Guid
  TenantId: Guid
  Slug: string
  Name: string
  Capacity: int
  PaymentMode: PaymentMode          # Manual | Automatic
  PaymentProvider: string?          # e.g. "Stripe", null for Manual
  IsDeleted: bool

  abstract ResolveSlot(requestedStart, tz) → (Start, End)
  abstract GetEffectiveRange(start, end)   → (EffectiveStart, EffectiveEnd)

sealed TimeSlotBookingType : BookingType
  DurationMinutes: int
  BufferBeforeMinutes: int           # shifts next available slot start
  BufferAfterMinutes: int            # shifts next available slot start
  AvailabilityWindows: IReadOnlyList<TimeSlotWindow>

sealed CalendarBookingType : BookingType
  AvailableDays: IReadOnlyList<DayOfWeek>
```

### 3.2 AvailabilityWindow

```
TimeSlotWindow                        # used by TimeSlotBookingType only
  DayOfWeek: DayOfWeek
  StartTime: TimeOnly                 # in tenant local time (IANA)
  EndTime: TimeOnly                   # in tenant local time (IANA)

CalendarBookingType uses IReadOnlyList<DayOfWeek> directly — no window entity needed.
```

In the DB, AvailabilityWindows table stores all windows. `StartTime`/`EndTime` are nullable — null for calendar windows. Infrastructure mapper projects accordingly.

### 3.3 Booking

```
Booking
  Id: Guid
  TenantId: Guid
  BookingTypeId: Guid
  Start: DateTimeOffset               # UTC
  End: DateTimeOffset                 # UTC
  Status: BookingStatus
  CustomerId: string                  # JWT sub claim at creation time
  CustomerEmail: string
  PaymentReference: string?           # gateway payment ID (Automatic mode)
  IsDeleted: bool
  RowVersion: uint                    # EF optimistic concurrency token
  StatusChanges: IReadOnlyList<BookingStatusChange>  # audit log (owned)

BookingStatus: enum
  PendingPayment | PendingVerification | Confirmed | Cancelled

BookingStatusChange                   # append-only owned entity
  Id: Guid
  BookingId: Guid
  FromStatus: BookingStatus
  ToStatus: BookingStatus
  ChangedById: string                 # JWT sub claim
  ChangedByRole: string               # TenantAdmin | TenantStaff | TenantPaymentService | Customer
  ChangedAt: DateTimeOffset
```

### 3.4 Tenant

```
Tenant
  Id: Guid
  Slug: string                        # unique, indexed
  Name: string
  TimeZoneId: string                  # IANA (e.g. "America/New_York")
  IsDeleted: bool
  CreatedAt: DateTimeOffset
```

### 3.5 Webhook

```
Webhook
  Id: Guid
  TenantId: Guid
  BookingTypeId: Guid
  Url: string
  Secret: string                      # HMAC signing secret
  IsDeleted: bool
```

### 3.6 Enums

```
BookingKind:   TimeSlot | Calendar
```

## 4. Booking State Machine

### State transitions

| From                  | Action  | Endpoint           | Allowed Roles                            |
| --------------------- | ------- | ------------------ | ---------------------------------------- |
| `PendingPayment`      | pay     | `POST .../pay`     | TenantAdmin, TenantStaff                 |
| `PendingVerification` | confirm | `POST .../confirm` | TenantAdmin, TenantStaff                 |
| `PendingPayment`      | cancel  | `POST .../cancel`  | TenantAdmin, TenantStaff, Customer (own) |
| `PendingVerification` | cancel  | `POST .../cancel`  | TenantAdmin, TenantStaff, Customer (own) |
| `Confirmed`           | cancel  | `POST .../cancel`  | TenantAdmin, TenantStaff, Customer (own) |
| `Cancelled`           | any     | —                  | `DomainException` (terminal state)       |

### Payment flows

**Manual (`PaymentMode = Manual`):**

```
POST /bookings → PendingPayment
POST /pay      → PendingVerification   (Admin/Staff)
POST /confirm  → Confirmed             (Admin/Staff)
```

**Automatic (`PaymentMode = Automatic`):**

```
POST /bookings          → PendingPayment + returns PaymentReference (stub UUID in v0.1)
POST /payments/webhook  → internal /pay transition (stubbed gateway handler)
POST /confirm           → Confirmed (Admin/Staff)
```

## 5. Booking Type Logic

### TimeSlot — slot generation

```
Window: Mon 09:00–17:00, DurationMinutes=60, BufferBefore=0, BufferAfter=15

Candidate slots (advancing by Duration + BufferBefore + BufferAfter):
  09:00–10:00
  10:15–11:15
  11:30–12:30
  ...
```

Customer submits `startTime` (UTC). End derived: `start + DurationMinutes`.

### TimeSlot — effective conflict range

Buffer shifts the next slot, not the stored booking bounds. Conflict SQL uses:

```
effectiveStart = existingStart - BufferBeforeMinutes
effectiveEnd   = existingEnd   + BufferAfterMinutes

condition: effectiveStart < newEnd AND effectiveEnd > newStart
```

### Calendar — slot generation

One slot per available day in the requested window. Day is available if:

- `DayOfWeek` is in `AvailableDays`
- `bookingCount(day) < Capacity` for the relevant statuses

Customer submits `date` string (e.g. `2026-03-06`). Converted to UTC day boundaries server-side using tenant `TimeZoneId`.

### Conflict SQL (both kinds — runs as DB query, never in memory)

```sql
SELECT COUNT(*)
FROM Bookings
WHERE BookingTypeId = @bookingTypeId
  AND (Start - @bufferBefore) < @newEnd
  AND (End + @bufferAfter)   > @newStart
  AND Status IN ('PendingPayment','PendingVerification','Confirmed')
  AND IsDeleted = false
  AND TenantId = @tenantId
```

If `COUNT >= Capacity` → `SlotConflictException` (HTTP 409).

## 6. Database Indexes

All indexes defined via EF Fluent API (provider-neutral).

| Table                 | Index                                   | Type       | Purpose                  |
| --------------------- | --------------------------------------- | ---------- | ------------------------ |
| `Tenants`             | `(Slug)`                                | Unique     | Slug resolution          |
| `BookingTypes`        | `(TenantId, Slug)`                      | Unique     | Slug lookup per tenant   |
| `BookingTypes`        | `(TenantId, IsDeleted)`                 | Non-unique | Filtered listing         |
| `Bookings`            | `(TenantId, BookingTypeId, Start, End)` | Non-unique | Overlap range query      |
| `Bookings`            | `(BookingTypeId, Status, Start, End)`   | Non-unique | Conflict check           |
| `Bookings`            | `(TenantId, IsDeleted)`                 | Non-unique | Soft-delete filter       |
| `Bookings`            | `(CustomerId)`                          | Non-unique | Customer ownership check |
| `AvailabilityWindows` | `(BookingTypeId)`                       | Non-unique | Window load              |
| `Webhooks`            | `(TenantId, BookingTypeId)`             | Non-unique | Webhook lookup           |

### Global query filters (EF)

```csharp
modelBuilder.Entity<BookingTypeEntity>()
    .HasQueryFilter(bt => !bt.IsDeleted && bt.TenantId == _tenantId);

modelBuilder.Entity<BookingEntity>()
    .HasQueryFilter(b => !b.IsDeleted && b.TenantId == _tenantId);

modelBuilder.Entity<WebhookEntity>()
    .HasQueryFilter(w => !w.IsDeleted && w.TenantId == _tenantId);
```

`_tenantId` resolved from scoped `ITenantContext` injected into `DbContext`.

## 7. Critical Query Patterns

### Availability (one DB round-trip)

```csharp
// 1. Fetch existing bookings in range — projection only, no entity tracking
var bookedSlots = await dbContext.Bookings
    .AsNoTracking()
    .Where(b => b.BookingTypeId == bookingTypeId
             && b.Start < to && b.End > from
             && conflictStatuses.Contains(b.Status))
    .Select(b => new { b.Start, b.End })
    .ToListAsync(ct);

// 2. Generate candidate slots in C# from weekly windows (bounded by from/to)
// 3. Subtract bookedSlots in memory, return available slots
```

### Booking list (paginated, no N+1)

```csharp
dbContext.Bookings
    .AsNoTracking()
    .Where(b => b.BookingTypeId == bookingTypeId)
    .OrderBy(b => b.Start)
    .Skip(pageSize * (page - 1))
    .Take(pageSize)
    .Select(b => new BookingListItemDto { ... })
    .ToListAsync(ct);
```

### Conflict check (COUNT query, no materialization)

```csharp
var conflictCount = await dbContext.Bookings
    .Where(b => b.BookingTypeId == bookingTypeId
             && (b.Start - bufferBefore) < newEnd
             && (b.End + bufferAfter) > newStart
             && conflictStatuses.Contains(b.Status))
    .CountAsync(ct);
```

### Slug lookup (composite index hit, no table scan)

```csharp
dbContext.BookingTypes
    .AsNoTracking()
    .Where(bt => bt.TenantId == tenantId && bt.Slug == slug)
    .Select(bt => new { bt.Id, bt.Kind, ... })
    .FirstOrDefaultAsync(ct);
```

## 8. Application Layer (CQRS via MediatR)

### Commands

- `CreateBookingCommand` — validates slot, conflict check, insert with optimistic concurrency retry (max 3)
- `PayBookingCommand` — PendingPayment → PendingVerification
- `ConfirmBookingCommand` — PendingVerification → Confirmed
- `CancelBookingCommand` — any → Cancelled (ownership check for Customer role)
- `CreateBookingTypeCommand`
- `UpdateBookingTypeCommand`
- `DeleteBookingTypeCommand` — soft delete
- `CreateWebhookCommand`
- `DeleteWebhookCommand` — soft delete

### Queries

- `GetAvailabilityQuery` — availability algorithm
- `GetBookingQuery`
- `ListBookingsQuery` — paginated
- `GetBookingTypeQuery`
- `ListBookingTypesQuery` — paginated
- `GetWebhooksQuery`
- `GetTenantQuery`

### Pipeline Behaviors (in order)

1. `LoggingBehavior<,>` — request/response duration
2. `ValidationBehavior<,>` — FluentValidation, throws on failure
3. `PerformanceBehavior<,>` — warns if handler exceeds 50ms (reads) / 75ms (writes)

### Tenant resolution

- `ITenantContext` in Application layer (interface only)
- `JwtTenantContext` in Infrastructure — reads `tenant_id` + `sub` + `role` from `IHttpContextAccessor`
- Registered as `Scoped`

## 10. Concurrency Control

`BookingType` carries a `RowVersion` (uint) EF concurrency token. `CreateBookingCommand` wraps conflict-check + insert in a transaction. On `DbUpdateConcurrencyException`, retries up to 3 times before throwing `SlotConflictException`. This is provider-portable (EF handles the token comparison).

## 11. Webhook Delivery (v0.1 stub)

Webhook entity CRUD is fully implemented. Delivery on state transitions is a stub:

- `BookingStatusChangedNotification` (MediatR `INotification`) is published after each status change
- `WebhookDeliveryHandler` handles the notification, logs intent, returns without HTTP call
- Real delivery wired in v0.2

Payment webhook receiver (`POST /payments/webhook`) is also stubbed — accepts call, validates HMAC signature stub, transitions `PendingPayment → PendingVerification`.

## 12. Timezone Rules

- Weekly windows stored in **tenant local time** (`TimeOnly` fields)
- All `Booking.Start`/`End` stored as **UTC** `DateTimeOffset`
- Conversion during slot generation using `TimeZoneInfo.FindSystemTimeZoneById(tenant.TimeZoneId)`
- IANA timezone IDs supported via `TimeZoneInfo` on .NET 10 (cross-platform)
- Tests must cover DST transitions (e.g. clocks spring forward, window boundary falls in gap)

## 14. Test Strategy

| Layer       | Tools                                        | DB                   | Scope                                                             |
| ----------- | -------------------------------------------- | -------------------- | ----------------------------------------------------------------- |
| Unit        | xUnit, FluentAssertions                      | None                 | Domain rules, conflict logic, slot generation, state machine, DST |
| Integration | xUnit, Testcontainers (Postgres)             | Real Postgres        | EF queries, indexes, migrations, transactions, concurrency        |
| Functional  | xUnit, WebApplicationFactory, Testcontainers | Real Postgres        | Full HTTP booking flows, auth, pagination                         |
| Performance | BenchmarkDotNet                              | In-memory / mocked   | Slot generation throughput, conflict query algorithm              |
| Load        | k6 (JS scripts)                              | Running Docker stack | Concurrent bookings, no double-booking, latency targets           |

### Response time targets

- Availability endpoint: < 100ms
- Booking creation: < 80ms
- Booking listing (paginated): < 100ms
- 10k booking conflict check: remains efficient (COUNT query, indexed)

### TDD order per feature

1. Unit test (domain rule, pure)
2. Integration test (EF query)
3. Functional test (HTTP endpoint)
4. Performance benchmark

## 15. Data Source Switchability

- All queries use EF Core LINQ — no raw SQL, no stored procedures, no provider-specific functions
- Provider configured via `appsettings.json`: `"Provider": "PostgreSQL"` (future: `"SqlServer"`, `"MySQL"`)
- `IDbProviderConfigurator` interface in Infrastructure with `PostgreSqlConfigurator` implementation
- Migrations per provider in `Migrations/{Provider}/` folders
- No `EF.Functions.xxx` provider-specific calls in query logic

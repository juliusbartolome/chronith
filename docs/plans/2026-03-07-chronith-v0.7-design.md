# Chronith v0.7 Design — API Maturity & Core Feature Gaps

**Date:** 2026-03-07
**Prerequisite:** v0.6 must be completed and tagged before v0.7 implementation begins.
**Branch:** `feat/v0.7-api-maturity` (from `develop`)

---

## Overview

v0.7 fills the remaining feature gaps and establishes API versioning so the dashboard (v0.9) builds against a stable contract. Four pillars, implemented in dependency order.

| Pillar | Theme              | Key Deliverables                                                        |
| ------ | ------------------ | ----------------------------------------------------------------------- |
| 1      | API Versioning     | `/v1/` route prefix on all endpoints, deprecation redirects             |
| 2      | Customer Accounts  | Registration, login, profile, booking history (built-in + OIDC)         |
| 3      | Recurring Bookings | Simple repeat rules (daily/weekly/monthly), series management           |
| 4      | Idempotency Keys   | Duplicate prevention for mutating requests via `Idempotency-Key` header |

## Pillar 1: API Versioning

Introduce a `/v1/` route prefix on all endpoints. This is a routing-only change — no domain model modifications.

### Approach

Use FastEndpoints route prefix grouping to move all existing routes under `/v1/`. Examples:

- `/booking-types` → `/v1/booking-types`
- `/bookings/{id}` → `/v1/bookings/{id}`
- `/public/{tenantSlug}/...` → `/v1/public/{tenantSlug}/...`
- `/health/live` and `/health/ready` remain unversioned (infrastructure endpoints).

### Deprecation Strategy

Unversioned routes return `308 Permanent Redirect` to their `/v1/` equivalents for one major version cycle. The redirects are removed in v1.0. 308 is used (not 301) because it preserves the HTTP method (POST, PUT, etc.) on redirect.

### Implementation

A single `RoutePrefix("v1")` group applied to all endpoint base classes, or a global prefix configured in `Program.cs`. Health check endpoints are excluded from the prefix.

### Testing

- Update all existing functional tests to use `/v1/` prefix.
- Add redirect tests: verify unversioned routes return 308 with correct `Location` header.

## Pillar 2: Customer Accounts

Customers currently have no accounts — public bookings use email-only identity. For a SaaS platform, customers need to register, log in, view their booking history, and manage their profile.

### Auth Model

Dual-mode per tenant:

- **Built-in auth** — email + password (BCrypt), optional magic link. Extends the existing auth infrastructure.
- **External OIDC** — delegate to Auth0, Clerk, Firebase Auth, etc. Tenant configures OIDC issuer, client ID, and audience. Chronith validates external JWTs.

Tenants configure which mode(s) are active via `TenantAuthConfig`.

### Domain Model

```
Customer
├── Id: Guid
├── TenantId: Guid
├── Email: string (unique per tenant)
├── PasswordHash: string? (null for OIDC-only customers)
├── Name: string
├── Phone: string?
├── ExternalId: string? (from OIDC provider)
├── AuthProvider: string ("builtin" | "oidc")
├── IsEmailVerified: bool
├── IsActive: bool
├── IsDeleted: bool
├── CreatedAt: DateTimeOffset
└── LastLoginAt: DateTimeOffset?
```

```
TenantAuthConfig
├── Id: Guid
├── TenantId: Guid
├── AllowBuiltInAuth: bool (default true)
├── OidcIssuer: string?
├── OidcClientId: string?
├── OidcAudience: string?
├── MagicLinkEnabled: bool (default false)
├── CreatedAt: DateTimeOffset
└── UpdatedAt: DateTimeOffset
```

### Booking Changes

`CustomerId` on `Booking` becomes a FK to the `Customer` entity (nullable for backwards compatibility with pre-v0.7 bookings that used string-based customer IDs).

A migration adds the `customers` table, `tenant_auth_configs` table, and the new FK column on `bookings`.

### Endpoints

| Method | Route                                | Access      | Description             |
| ------ | ------------------------------------ | ----------- | ----------------------- |
| POST   | `/v1/public/{slug}/auth/register`    | Anonymous   | Customer registration   |
| POST   | `/v1/public/{slug}/auth/login`       | Anonymous   | Customer login (JWT)    |
| POST   | `/v1/public/{slug}/auth/refresh`     | Customer    | Refresh token           |
| GET    | `/v1/public/{slug}/auth/me`          | Customer    | Customer profile        |
| PUT    | `/v1/public/{slug}/auth/me`          | Customer    | Update profile          |
| GET    | `/v1/public/{slug}/my-bookings`      | Customer    | Booking history         |
| GET    | `/v1/public/{slug}/my-bookings/{id}` | Customer    | Single booking detail   |
| PUT    | `/v1/tenant/auth-config`             | TenantAdmin | Configure auth mode     |
| GET    | `/v1/tenant/auth-config`             | TenantAdmin | Get current auth config |

### Registration Flow

1. Customer sends `POST /v1/public/{slug}/auth/register` with `{ email, password, name }`.
2. Server validates email uniqueness within the tenant, hashes password (BCrypt), creates `Customer` record.
3. If tenant has `MagicLinkEnabled`, send verification email instead of requiring password.
4. Returns JWT (with `role: "Customer"`, `tenant_id`, `customer_id` claims).

### OIDC Flow

1. Customer authenticates with the external IdP (client-side).
2. Customer sends the IdP's `id_token` to `POST /v1/public/{slug}/auth/login` with `{ provider: "oidc", token: "<id_token>" }`.
3. Server validates the token against the tenant's configured OIDC issuer/audience.
4. If no `Customer` record exists for this `ExternalId`, one is created automatically.
5. Returns a Chronith JWT.

### Constraints

- A customer email must be unique within a tenant.
- Built-in and OIDC customers can coexist within the same tenant.
- Deactivating a customer does not cancel their existing bookings.
- Customer JWTs include `customer_id` claim so booking ownership can be verified.

## Pillar 3: Recurring Bookings

Simple repeat rules first. The model is extensible to full RRULE (RFC 5545) in a future version.

### Domain Model

```
RecurrenceRule
├── Id: Guid
├── TenantId: Guid
├── BookingTypeId: Guid
├── CustomerId: Guid
├── StaffMemberId: Guid? (optional preference)
├── Frequency: RecurrenceFrequency (Daily, Weekly, Monthly)
├── Interval: int (every N frequencies — e.g., every 2 weeks)
├── DaysOfWeek: int[]? (for weekly: [1,3] = Mon,Wed — stored as JSONB)
├── StartTime: TimeOnly
├── Duration: TimeSpan
├── SeriesStart: DateOnly (first occurrence)
├── SeriesEnd: DateOnly? (null = no end)
├── MaxOccurrences: int? (alternative to end date)
├── IsActive: bool
├── IsDeleted: bool
├── CreatedAt: DateTimeOffset
```

```csharp
public enum RecurrenceFrequency { Daily, Weekly, Monthly }
```

### Booking Changes

Add `RecurrenceRuleId: Guid?` on `Booking`. Individual occurrences are standard `Booking` records linked to their series via this FK. This preserves all existing booking behavior (state machine, payments, staff assignment) for each occurrence.

### Generation Strategy

A background service (`RecurringBookingGeneratorService`) runs daily:

1. Query all active `RecurrenceRule` records.
2. For each rule, compute the next N days of occurrences (configurable horizon, default 30 days).
3. For each computed occurrence date/time, check if a `Booking` already exists for that `(RecurrenceRuleId, Start)`.
4. If not, validate availability (including staff availability if assigned) and create the booking.
5. If a slot is unavailable, skip that occurrence and log a warning (do not fail the series).

**Configuration:**

```
RecurringBookings:GenerationHorizonDays = 30
RecurringBookings:CheckIntervalHours = 24
```

### Series Management

- **Cancel single occurrence** — cancels one booking via existing `POST /v1/bookings/{id}/cancel`. The series continues.
- **Cancel series** — sets `RecurrenceRule.IsActive = false`, cancels all future bookings in status `PendingPayment` or `PendingVerification`. Confirmed bookings are left as-is.
- **Modify series** — updates the rule (time, frequency, staff). Regenerates future unconfirmed bookings on the next generator run.
- **Skip occurrence** — cancel a specific future occurrence without affecting the series. The generator won't recreate it (tracked by a `SkippedDate` concept or by checking for cancelled bookings at that time).

### Endpoints

| Method | Route                                      | Access                             | Description               |
| ------ | ------------------------------------------ | ---------------------------------- | ------------------------- |
| POST   | `/v1/booking-types/{slug}/recurring`       | TenantAdmin, TenantStaff, Customer | Create recurring series   |
| GET    | `/v1/recurring`                            | TenantAdmin, TenantStaff           | List all recurrence rules |
| GET    | `/v1/recurring/{id}`                       | TenantAdmin, TenantStaff, Customer | Get rule + upcoming       |
| PUT    | `/v1/recurring/{id}`                       | TenantAdmin, TenantStaff           | Modify series             |
| DELETE | `/v1/recurring/{id}`                       | TenantAdmin, TenantStaff, Customer | Cancel series             |
| GET    | `/v1/recurring/{id}/occurrences?from=&to=` | TenantAdmin, TenantStaff, Customer | List occurrences in range |

### Availability Impact

When computing availability, the generator's pre-created bookings already occupy slots. No special availability logic is needed — the existing availability query naturally excludes slots with bookings.

### Constraints

- A recurrence rule must reference a valid booking type and customer.
- If `RequiresStaffAssignment` is true on the booking type, `StaffMemberId` must be provided on the rule.
- `SeriesEnd` or `MaxOccurrences` — at least one must be set to prevent unbounded series (configurable max: 365 days or 52 occurrences).
- The generator respects time blocks and staff availability — it skips unavailable slots.

## Pillar 4: Idempotency Keys

Prevent duplicate booking creation and payment processing from client retries or network failures.

### Implementation

**Header:** `Idempotency-Key: <uuid>` on mutating requests (POST).

**Storage:**

```
IdempotencyKey
├── Id: Guid
├── TenantId: Guid
├── Key: string (the client-provided UUID)
├── EndpointRoute: string
├── RequestHash: string (SHA-256 of request body)
├── ResponseStatusCode: int
├── ResponseBody: string (JSONB)
├── CreatedAt: DateTimeOffset
├── ExpiresAt: DateTimeOffset (CreatedAt + 24h)
```

**Flow:**

1. Client sends a POST request with `Idempotency-Key` header.
2. FastEndpoints pre-processor checks for existing key in the `idempotency_keys` table.
3. If found and not expired:
   - If `RequestHash` matches, return the cached response (same status code + body).
   - If `RequestHash` differs, return `422 Unprocessable Entity` — same key with different payload is an error.
4. If not found, proceed with normal handling. After the response is generated, store the key + response.
5. If no `Idempotency-Key` header is present, the request proceeds normally (idempotency is opt-in).

**Applicable Endpoints:**

- `POST /v1/booking-types/{slug}/bookings` — create booking
- `POST /v1/bookings/{id}/pay` — mark as paid
- `POST /v1/bookings/{id}/confirm` — confirm booking
- `POST /v1/booking-types/{slug}/recurring` — create recurring series
- `POST /v1/waitlist/{id}/accept` — accept waitlist offer
- `POST /v1/public/{slug}/auth/register` — customer registration

**Cleanup:** A background task or database TTL (PostgreSQL `pg_cron` or application-level) purges expired keys.

### Constraints

- Keys are scoped per tenant — two tenants can use the same UUID without collision.
- Unique constraint on `(TenantId, Key, EndpointRoute)`.
- Max key length: 128 characters.
- Expiration: 24 hours (configurable via `Idempotency:ExpirationHours`).

## Cross-Cutting Concerns

### Database Migration

One migration covering all new tables and columns:

- `customers` table
- `tenant_auth_configs` table
- `customer_id` FK column on `bookings` (nullable)
- `recurrence_rules` table
- `recurrence_rule_id` FK column on `bookings` (nullable)
- `idempotency_keys` table

### Testing Strategy

- **Unit tests** for all new domain models: `Customer` (create, update, verify email, deactivate), `RecurrenceRule` (create, compute occurrences, cancel series, modify), idempotency key matching logic.
- **Integration tests** for customer repository, recurrence rule repository, idempotency key storage/retrieval.
- **Functional tests** for all new endpoints: customer auth flows (register, login, refresh, OIDC), recurring CRUD, idempotency replay, `/v1/` prefix migration.
- **Test coverage target:** >90% for new code in v0.7.
- **Regression:** All existing tests updated for `/v1/` prefix and passing.

### Dependency Order

```
Pillar 1 (API Versioning)
    ↓
Pillar 2 (Customer Accounts) — endpoints use /v1/ prefix
    ↓
Pillar 3 (Recurring Bookings) — references Customer entity
    ↓
Pillar 4 (Idempotency Keys) — applies to endpoints from Pillars 2 and 3
```

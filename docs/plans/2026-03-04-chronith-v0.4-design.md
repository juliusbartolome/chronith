# Chronith v0.4 — Architecture Design Document

**Status:** Approved
**Date:** 2026-03-04
**Theme:** Tenant Self-Service, Scalability & Customer Notifications
**Prerequisite:** v0.3 fully implemented (`WebhookOutboxEntries` API, per-tenant rate limiting, OpenAPI, tenant metrics in place)

---

## Overview

v0.4 addresses three pillars delivered as sequential feature branches, each reviewed independently on `develop`:

1. **Pillar 1 — Full Auth (`feature/auth-user-accounts`)** — `TenantUser` entity, `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `GET/PATCH /auth/me`; makes Chronith self-serve without direct DB access
2. **Pillar 2 — Redis Infra (`feature/redis-infra`)** — Redis-backed rate limiter (`RedisRateLimitStore`) + Redis caching for metrics and availability endpoints
3. **Pillar 3 — Customer Callbacks (`feature/customer-callbacks`)** — per-`BookingType` customer callback URL, dispatched via the existing webhook outbox infrastructure

Breaking changes are permitted in v0.4.

## Pillar 1 — Full Auth

### New Domain Entity: `TenantUser`

Defined in `Chronith.Domain/Models/TenantUser.cs`:

```
TenantUser
  Id           : Guid (PK)
  TenantId     : Guid (FK → Tenant)
  Email        : string (unique per tenant)
  PasswordHash : string (BCrypt, cost factor 12)
  Role         : TenantUserRole enum { Owner, Admin, Member }
  CreatedAt    : DateTimeOffset
  IsActive     : bool
```

Users are always scoped to a tenant — there is no global `User` entity. The first registered user for a tenant is automatically assigned the `Owner` role. Role is included in JWT claims for extensibility; per-role endpoint authorization is deferred to a future version.

### New Domain Entity: `TenantUserRefreshToken`

Defined in `Chronith.Domain/Models/TenantUserRefreshToken.cs`:

```
TenantUserRefreshToken
  Id          : Guid (PK)
  TenantUserId: Guid (FK → TenantUser)
  TokenHash   : string (SHA-256 hash of the opaque Guid token)
  ExpiresAt   : DateTimeOffset (30-day TTL)
  UsedAt      : DateTimeOffset? (null = still valid)
  CreatedAt   : DateTimeOffset
```

Refresh tokens are single-use and rotated on each refresh call.

### New Endpoints

| Method   | Path               | Auth Required | Description                              |
| -------- | ------------------ | ------------- | ---------------------------------------- |
| `POST`   | `/auth/register`   | None          | Register new tenant + owner account      |
| `POST`   | `/auth/login`      | None          | Exchange credentials for JWT + refresh token |
| `POST`   | `/auth/refresh`    | None          | Issue new JWT using refresh token        |
| `GET`    | `/auth/me`         | JWT Bearer    | Return current user profile              |
| `PATCH`  | `/auth/me`         | JWT Bearer    | Update email or password                 |

### Token Design

- **Access token**: JWT, 15-minute expiry, signed with existing symmetric HMAC key. Claims: `sub` (TenantUserId), `tenant_id`, `email`, `role`
- **Refresh token**: opaque `Guid`, stored hashed (`SHA-256`) in `TenantUserRefreshTokens`, 30-day expiry, single-use (rotated on each use)
- **No change to existing `X-Api-Key` flow** — API keys remain for machine-to-machine; JWTs are for human/dashboard access

### `POST /auth/register` Behaviour

- Creates a new `Tenant` (name from request body) and a `TenantUser` with `Owner` role in a single transaction
- Password constraints: minimum 8 characters, at least 1 digit
- Returns access token + refresh token immediately (no email verification in v0.4)
- Duplicate email returns `409 Conflict` with Problem Details

### Updated Auth Matrix

Existing endpoints accept **either** API key or JWT Bearer — no breaking change to existing callers.

| Endpoint group                                                | API Key | JWT Bearer | Anonymous |
| ------------------------------------------------------------- | :-----: | :--------: | :-------: |
| `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh` |   —   |     —      |     ✓     |
| `GET /auth/me`, `PATCH /auth/me`                              |    —    |     ✓      |     —     |
| All existing tenant/booking endpoints                         |    ✓    |     ✓      |     —     |

## Pillar 2 — Redis Infra

### Redis Rate Limiter

The `IRateLimitStore` abstraction introduced in v0.3 enables a drop-in swap. The new implementation:

```
RedisRateLimitStore : IRateLimitStore
  - Uses StackExchange.Redis IConnectionMultiplexer
  - Lua script for atomic sliding-window increment + expiry (same semantics as InMemoryRateLimitStore)
  - Key format: ratelimit:{tenantId}:{endpoint}:{windowStart}
  - Falls back gracefully to InMemoryRateLimitStore if Redis is unavailable (Polly circuit breaker)
```

`InMemoryRateLimitStore` is retained as the fallback — no existing code is deleted.

DI registration logic: if `IConnectionMultiplexer` resolves successfully → register `RedisRateLimitStore`; otherwise → register `InMemoryRateLimitStore`. Controlled by `Redis:Enabled` config flag.

### Redis Caching

New `IRedisCacheService` interface in `Chronith.Application/Interfaces/`:

```csharp
public interface IRedisCacheService
{
    Task<T?> GetOrSetAsync<T>(string key, Func<Task<T>> factory, TimeSpan ttl, CancellationToken ct = default);
    Task InvalidateAsync(string key, CancellationToken ct = default);
}
```

Backed by `IDistributedCache` (StackExchange.Redis provider). Values serialized as JSON via `System.Text.Json`. Cache miss = normal handler execution; result written to cache before returning.

Cache targets:

| Endpoint | Cache Key | TTL | Invalidation Trigger |
| -------- | --------- | --- | -------------------- |
| `GET /tenant/metrics` | `metrics:{tenantId}` | 60 s | Any booking state change for that tenant |
| `GET /booking-types/{id}/availability` | `availability:{bookingTypeId}:{date}` | 30 s | Any booking created or cancelled for that booking type |

No stale-while-revalidate — simple read-through / write-through is sufficient for v0.4.

### Docker Compose

Redis 8 (Alpine) added to `docker-compose.yml`:

```yaml
redis:
  image: redis:8-alpine
  ports:
    - "6379:6379"
  healthcheck:
    test: ["CMD", "redis-cli", "ping"]
    interval: 5s
    timeout: 3s
    retries: 5
```

The app service `depends_on` Redis with `condition: service_healthy` (same pattern as Postgres).

### Configuration

New `Redis` section in `appsettings.json`:

```json
"Redis": {
  "ConnectionString": "localhost:6379",
  "Enabled": true
}
```

`Enabled: false` keeps the in-memory fallback for local dev without Docker.

## Pillar 3 — HTTP Customer Callbacks

### Domain Change: `BookingType`

Add one nullable field to the existing `BookingType` abstract entity:

```
CustomerCallbackUrl : string? (nullable; validated as HTTPS URL when present)
```

Exposed via the existing `PATCH /booking-types/{id}` endpoint as a new optional field. No new endpoint required.

### Outbox Category Discriminator

New `OutboxCategory` enum added to `Chronith.Domain`:

```csharp
public enum OutboxCategory { TenantWebhook, CustomerCallback }
```

`WebhookOutboxEntry` gains a `Category` column (non-nullable, default `TenantWebhook`). Existing rows are backfilled as `TenantWebhook` in the migration.

### New Event Types

| Domain Event | Event Type String | Category |
| ------------ | ---------------- | -------- |
| `BookingConfirmedDomainEvent` | `customer.booking.confirmed` | `CustomerCallback` |
| `BookingCancelledDomainEvent` | `customer.booking.cancelled` | `CustomerCallback` |
| `PaymentReceivedDomainEvent` | `customer.payment.received` | `CustomerCallback` |

Existing `TenantWebhook` entries for `booking.confirmed` and `booking.cancelled` continue unchanged.

Customer callback payload:

```json
{
  "bookingId": "uuid",
  "bookingReference": "CHR-20260304-XXXX",
  "customerEmail": "customer@example.com",
  "eventType": "customer.booking.confirmed",
  "occurredAt": "2026-03-04T10:00:00Z"
}
```

No sensitive payment data included.

### Dual Outbox Write

On booking state changes, two outbox entries are written in the same transaction:

- `BookingConfirmedDomainEvent` → writes a `TenantWebhook` entry (existing) **and** a `CustomerCallback` entry (new)
- `BookingCancelledDomainEvent` → same dual write
- `PaymentReceivedDomainEvent` → writes only a `CustomerCallback` entry (no tenant webhook today)

### Dispatcher Reuse

`WebhookDispatcherService` is extended minimally — the batch query is updated to fetch entries across both `OutboxCategory` values:

- For `TenantWebhook` entries: destination URL = `Webhook.Url` (existing behaviour)
- For `CustomerCallback` entries: destination URL = `BookingType.CustomerCallbackUrl` looked up at dispatch time; if URL is null or removed, entry is marked `Abandoned` (not retried)
- Same HMAC-SHA256 signing, same retry schedule (30 s / 2 m / 10 m / 1 h / 4 h, max 6 attempts)
- No separate dispatcher service — one poller, two categories

### Clean Architecture Placement

| Component | Project |
| --------- | ------- |
| `OutboxCategory` enum | `Chronith.Domain/Enums/` |
| `CustomerCallbackUrl` on `BookingType` | `Chronith.Domain/Models/BookingType.cs` |
| `AddCustomerCallbackUrl` EF migration | `Chronith.Infrastructure/Migrations/PostgreSQL/` |
| `AddOutboxCategory` EF migration | `Chronith.Infrastructure/Migrations/PostgreSQL/` |
| Updated `WebhookDispatcherService` | `Chronith.Infrastructure/Services/WebhookDispatcherService.cs` |

## DB Migrations Summary

Three new migrations, applied sequentially:

| Migration | Changes |
| --------- | ------- |
| `AddTenantUsers` | New `TenantUsers` and `TenantUserRefreshTokens` tables |
| `AddCustomerCallbackUrl` | Nullable `CustomerCallbackUrl` column on `BookingTypes` |
| `AddOutboxCategory` | `Category` column on `WebhookOutboxEntries`; backfill existing rows as `TenantWebhook` (0 = default) |

All existing data is preserved. No existing columns are modified or dropped.

| Layer | Scope |
| ----- | ----- |
| Unit | `TenantUser` domain rules (password length, role assignment), BCrypt hash round-trip, `RedisRateLimitStore` (mocked `IConnectionMultiplexer`), `IRedisCacheService` get/set/invalidate (mocked), dual outbox write on domain events, `Abandoned` status when `CustomerCallbackUrl` is null |

| Functional | End-to-end: register → login → create booking type with `CustomerCallbackUrl` → confirm booking → assert `CustomerCallback` outbox entry created with correct payload; availability cache hit after first call; metrics cache invalidated after booking state change |

## Branch Strategy

```
feature/auth-user-accounts   → develop   (Pillar 1)
feature/redis-infra          → develop   (Pillar 2)
feature/customer-callbacks   → develop   (Pillar 3)
```

Each branch requires a passing PR (`develop` branch protection: `.NET Tests` + `Docker Build` status checks + 1 approval). All three are merged before the v0.4 `develop` → `main` promotion.

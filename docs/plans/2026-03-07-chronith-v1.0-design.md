## Pillar 1: Complete Admin Dashboard

Build on v0.9's foundation to add all remaining dashboard pages.

### Analytics Dashboard (`/analytics`)

**Bookings Analytics Tab:**

- Line/bar chart: bookings over time (day/week/month granularity).
- Pie chart: bookings by status.
- Bar chart: bookings by booking type.
- Bar chart: bookings by staff member.
- KPI cards: total bookings, confirmed rate, cancellation rate, average bookings per day.
- Date range picker + groupBy selector.
- Export CSV/PDF buttons.

**Revenue Analytics Tab:**

- Line chart: revenue over time.
- Bar chart: revenue by booking type.
- Bar chart: revenue by staff member.
- KPI cards: total revenue, average booking value, revenue per staff member.
- Currency formatting per tenant.
- Date range picker + groupBy selector.
- Export buttons.

**Utilization Analytics Tab:**

- Heatmap or stacked bar: slot fill rates by booking type.
- Bar chart: utilization by staff member.
- KPI cards: overall utilization rate, busiest day of week, busiest time slot.
- Date range picker.

**Implementation:** All charts use Recharts. Data fetched via TanStack Query hooks calling the analytics SDK methods. Caching via React Query's stale-while-revalidate pattern (5-minute stale time matching the Redis cache TTL on the API).

### Notification Management (`/notifications`)

**Channel Configuration Page:**

- Cards for each channel (Email, SMS, Push).
- Each card: enabled/disabled toggle, status indicator (configured/not configured), configure button.
- Configuration forms per channel:
  - Email: SMTP host, port, username, password (masked), from address, test button.
  - SMS: Twilio Account SID, Auth Token (masked), From Number, test button.
  - Push: Firebase project ID, service account JSON upload, test button.
- Test button sends a test notification to the admin's email/phone.

**Template Editor Page:**

- List of templates grouped by event type.
- Click to edit: opens rich text editor.
- Variable picker sidebar: click a variable to insert `{{variable}}` at cursor.
- Live preview panel: shows rendered template with sample data (updates as you type).
- Channel tabs: switch between email (subject + body), SMS (body only), push (body only).
- Reset to default button per template.

**Notification Log Page:**

- Data table: recipient, channel, event type, status (delivered/failed/pending), sent at, actions (view details, retry).
- Filter by channel, status, date range.
- Detail modal: full notification content, delivery attempts with timestamps and error messages.

### Webhook Management (`/booking-types/[slug]` expanded)

- Enhanced webhook section on booking type detail page.
- List: URL, events subscribed (badges), status (active/failing), last delivery status.
- Create dialog: URL, event selection (checkboxes), optional secret.
- Detail view: delivery log with request headers, response status, response body (truncated), timestamps.
- Retry button for failed deliveries.
- Test button: sends a test webhook event.

### Customer Management (`/customers`)

**List Page:**

- Data table: Name, Email, Auth Provider (badge: built-in/OIDC), Bookings (count), Last Booking (date), Status (active/inactive), Actions (view, deactivate).
- Search by name or email.
- Filter by auth provider, status.

**Detail Page:**

- Customer profile: name, email, phone, auth provider, email verified, created date, last login.
- Booking history: paginated table of all customer bookings with status, date, type.
- Waitlist entries: active waitlist entries for this customer.
- Actions: deactivate (prevents new bookings), reactivate.

### Audit Log Viewer (`/audit`)

**List Page:**

- Data table: Timestamp, User (name + role badge), Entity Type, Entity ID (link), Action (badge), Summary.
- Filters: entity type (dropdown), action (dropdown), user (search), date range.
- Full-text search across metadata.

**Detail Modal:**

- Side-by-side diff view: old values (left, red highlights for removed) vs new values (right, green highlights for added).
- JSON tree view with expandable nodes.
- Metadata: IP address, user agent, correlation ID.

### Recurring Bookings (`/recurring`)

**List Page:**

- Data table: Customer, Booking Type, Frequency (badge: Daily/Weekly/Monthly), Next Occurrence, Total Occurrences, Status (active/cancelled), Actions (view, cancel series).
- Filter by booking type, status, customer.

**Detail Page:**

- Rule info: frequency, interval, days of week, start time, duration, series start/end.
- Customer and staff info.
- Upcoming occurrences: table of next N bookings generated from this rule.
- Past occurrences: paginated table of historical bookings.
- Actions: modify series (opens form), cancel series (confirm dialog), cancel individual occurrence.

### Features

**Booking Flow:**

1. Landing page: booking type cards showing name, description, duration, price. Tenant logo and primary color applied.
2. Date/time selection: calendar view for month navigation, available slots as clickable chips. Grayed-out dates = no availability.
3. Staff selection (if `RequiresStaffAssignment`): staff cards with name, photo placeholder, available times.
4. Customer details: name, email, phone (if logged in, pre-filled). Custom fields rendered dynamically from schema.
5. Review: summary of booking (type, date/time, staff, custom fields, price). "Confirm" button.
6. Payment (if applicable): embedded payment form for PayMongo/Maya. Redirect flow or inline widget depending on provider.
7. Confirmation: booking confirmed message, booking ID, "Add to Calendar" links (Google Calendar, Apple Calendar, Outlook — generated `.ics` download).

**Guest vs Authenticated:**

- Guests can book without creating an account (email required).
- "Login to see your booking history" link on the landing page.
- Authenticated customers see their booking history and can cancel/reschedule.

**Booking Lookup:**

- Guests can look up existing bookings by entering their email. Sends a one-time link to the email (magic link pattern) for security.

## Pillar 3: Tenant Self-Service Onboarding

### Signup Flow

1. **Landing page** (`/signup`): business name, admin email, password, timezone (auto-detected), country.
2. **Email verification**: send verification link, verify on click.
3. **Plan selection**: display available plans with feature comparison matrix.
4. **Payment setup** (if paid plan): payment form using existing payment providers (extended for subscriptions).
5. **Dashboard redirect**: first-time onboarding wizard starts.

### Onboarding Wizard

After signup, the admin lands on a guided wizard:

1. **Create first booking type**: name, kind, duration, availability windows (pre-filled with business hours).
2. **Set up notifications** (optional): connect email at minimum.
3. **Invite staff** (optional): add team members.
4. **Enable public booking**: toggle on, review public booking URL.
5. **Done**: dashboard with "Getting Started" checklist (dismissable).

### Billing Integration

**Extend `IPaymentProvider` with subscription methods:**

```csharp
public interface ISubscriptionProvider
{
    Task<SubscriptionResult> CreateSubscriptionAsync(
        CreateSubscriptionRequest request, CancellationToken ct);
    Task<SubscriptionResult> UpdateSubscriptionAsync(
        string subscriptionId, UpdateSubscriptionRequest request, CancellationToken ct);
    Task CancelSubscriptionAsync(
        string subscriptionId, CancellationToken ct);
    Task<SubscriptionStatusResult> GetSubscriptionStatusAsync(
        string subscriptionId, CancellationToken ct);
}
```

**Implementations:** Extend existing `PayMongoProvider` and `MayaProvider` with subscription support. Add a `StubSubscriptionProvider` for development/testing.

**Billing Webhook Receiver:**

- `POST /v1/webhooks/subscription` — receives subscription lifecycle events from the payment provider.
- Events: `subscription.paid`, `subscription.payment_failed`, `subscription.cancelled`.
- Updates `TenantSubscription.Status` accordingly.
- On `payment_failed`: set status to `PastDue`, send notification to tenant admin.
- On `PastDue` for > 7 days: set status to `Expired`, disable tenant API access (read-only mode).

### Plan Enforcement

**MediatR Pipeline Behavior:** `PlanEnforcementBehavior<TRequest, TResponse>` intercepts resource-creation commands:

1. Load tenant's active subscription and plan.
2. Check resource counts against plan limits.
3. If limit exceeded: throw `PlanLimitExceededException` with details (current count, limit, upgrade URL).
4. API returns 403 with Problem Details including the plan limit info.

**Feature Gating:**

- Notification endpoints: return 403 if `NotificationsEnabled = false` on the plan.
- Analytics endpoints: return 403 if `AnalyticsEnabled = false`.
- Custom branding: return 403 on `PUT /v1/tenant/settings` if branding fields changed and `CustomBrandingEnabled = false`.

**Usage Tracking:**

- Redis counters: `tenant:{id}:bookings:{yyyy-mm}` incremented on booking creation.
- Reconciled from database daily by a background service.
- `GET /v1/tenant/usage` returns current counts vs limits.

### Dashboard Integration

- **Settings > Subscription** page: current plan, usage meters (progress bars), upgrade/downgrade buttons, billing history.
- **Plan limit warnings**: toast notifications when approaching limits (80% threshold).
- **Upgrade prompts**: when a 403 plan limit error is received, show a contextual "Upgrade your plan" dialog.

### SDK Features

- **DI Integration:** `services.AddChronithClient(options => { options.BaseUrl = "..."; options.ApiKey = "..."; })`.
- **IHttpClientFactory:** uses named HTTP client for connection pooling and lifecycle management.
- **Auth:** supports both JWT bearer and API key authentication.
- **Retry policy:** configurable retry with exponential backoff for transient failures (via Polly or built-in).
- **Cancellation:** all methods accept `CancellationToken`.
- **Idempotency:** optional `idempotencyKey` parameter on mutating methods.

### CI Integration

- GitHub Actions job: regenerate on API changes, run tests, publish to NuGet (or GitHub Packages).
- Version tracks API version: `1.0.x`.

## Pillar 6: GA Readiness

### Security Audit

**Dependency Scan:**

```bash
dotnet list package --vulnerable
npm audit (dashboard + SDK)
```

- Fix all high and critical vulnerabilities.
- Document accepted risks for any low/medium that can't be fixed.

**OWASP Top 10 Review:**

| Category                  | Check                                                                 |
| ------------------------- | --------------------------------------------------------------------- |
| Injection                 | All inputs validated via FluentValidation, parameterized queries (EF) |
| Broken Authentication     | JWT validation, password hashing (BCrypt), rate limiting on auth      |
| Sensitive Data Exposure   | No PII in logs, credentials encrypted at rest, HTTPS enforced         |
| XML External Entities     | N/A (JSON-only API)                                                   |
| Broken Access Control     | Role-based endpoint auth, tenant isolation via query filters          |
| Security Misconfiguration | CSP headers, CORS, secure defaults                                    |
| Cross-Site Scripting      | HTML sanitization on inputs, CSP on dashboard                         |
| Insecure Deserialization  | JSON depth limits, max payload sizes                                  |
| Known Vulnerabilities     | Dependency scan (above)                                               |
| Insufficient Logging      | Audit logging, structured logs, correlation IDs                       |

**Penetration Test Checklist for Public Endpoints:**

- Verify tenant isolation: Tenant A cannot access Tenant B's data.
- Verify customer isolation: Customer A cannot access Customer B's bookings.
- Test rate limiting: verify limits enforced on auth and public endpoints.
- Test IDOR: verify all entity access checks tenant ownership.
- Test JWT manipulation: expired tokens, tampered claims, wrong signing key.

### Performance Pass

**Load Testing at Scale:**

- Target: 100 concurrent tenants, 1,000 requests/second sustained for 10 minutes.
- k6 scripts for new features:
  - `customer-auth-flow.js` — register, login, list bookings, refresh token.
  - `recurring-booking-generation.js` — create recurring rules, verify generation.
  - `analytics-queries.js` — concurrent analytics queries across tenants.
  - `public-booking-flow.js` — full public booking flow (browse, availability, book).
  - `dashboard-simulation.js` — simulate admin dashboard usage patterns.

**Bottleneck Identification:**

- Review OpenTelemetry traces for slow spans.
- Review PostgreSQL `pg_stat_statements` for slow queries.
- Check connection pool utilization under load.
- Verify Redis cache hit rates.

**Optimization:**

- Apply materialized views if analytics queries are slow.
- Add response caching for public endpoints (availability, booking types) with short TTL.
- Review and tune connection pool sizes.
- Consider read replica for analytics if needed.

### Test Coverage Audit

**Targets:**

| Layer          | Coverage Target | Tool              |
| -------------- | --------------- | ----------------- |
| Domain         | >95%            | coverlet + xUnit  |
| Application    | >95%            | coverlet + xUnit  |
| Infrastructure | >85%            | coverlet + xUnit  |
| API            | >80%            | coverlet + xUnit  |

**Edge Case Review:**

- Concurrent booking conflicts (optimistic concurrency).
- Expired waitlist offers during concurrent accept attempts.
- Timezone boundary conditions (DST transitions, UTC offset changes).
- Leap year dates for recurring bookings.
- Maximum payload sizes (custom fields, notification templates).
- Plan limit enforcement under concurrent requests.
- Token refresh race conditions.
- Idempotency key collision scenarios.

**Playwright E2E Coverage:**

- Every user flow in the admin dashboard has at least one E2E test.
- Every step of the public booking flow has an E2E test.
- Onboarding wizard (signup, plan selection, first booking type creation).
- All E2E tests pass on both desktop and mobile viewports.

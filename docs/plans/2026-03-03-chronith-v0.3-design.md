# Chronith v0.3 — Architecture Design Document

**Status:** Approved
**Date:** 2026-03-03
**Theme:** Observability & Ops
**Prerequisite:** v0.2 fully implemented (`WebhookOutboxEntries` table exists, API key auth in place)

---

## Overview

v0.3 adds four additive pillars with zero breaking changes to existing endpoints:

1. **Webhook Delivery Logs + Manual Retry** — expose `WebhookOutboxEntries` via API
2. **Per-Tenant Rate Limiting** — sliding window, in-memory, partitioned by `tenant_id` claim
3. **OpenAPI / Swagger Documentation** — FastEndpoints.Swagger, served at `/docs`
4. **Tenant Usage Metrics** — aggregate `GET /tenant/metrics` endpoint

All pillars are purely additive. No existing commands, domain models, or DB schema are modified beyond the single `RetryRequestedAt` column addition in Pillar 1.

## Pillar 1 — Webhook Delivery Logs + Manual Retry

### New Endpoints

| Method | Path                                           | Role                         |
| ------ | ---------------------------------------------- | ---------------------------- |
| `GET`  | `/webhooks/{id}/deliveries`                    | `TenantAdmin`, `TenantStaff` |
| `POST` | `/webhooks/{id}/deliveries/{deliveryId}/retry` | `TenantAdmin`                |

### GET /webhooks/{id}/deliveries

Paginated (`page`, `pageSize` query params, default 20, max 100). Sorted by `CreatedAt DESC`.

Response item shape:

```json
{
  "id": "uuid",
  "attempt_count": 3,
  "status": "Failed",
  "last_http_status": 500,
  "last_response_body": "Internal Server Error",
  "created_at": "2026-03-03T08:00:00Z",
  "last_attempted_at": "2026-03-03T10:15:00Z",
  "next_retry_at": null
}
```

Authorization: tenant ownership of the webhook is verified before listing. Returns `404` if the webhook does not belong to the caller's tenant (same pattern as all other tenant-scoped reads — enforced by EF global query filter on the `Webhook` lookup).

### POST /webhooks/{id}/deliveries/{deliveryId}/retry

Only valid for entries with `OutboxStatus.Failed`. Any other status returns `422 Unprocessable Entity` with a Problem Details body.

Action: reset `AttemptCount = 0`, `Status = Pending`, `NextRetryAt = UtcNow`, `RetryRequestedAt = UtcNow`.

Returns `204 No Content` on success.

### Domain Change: `WebhookOutboxEntry`

Add one nullable column: `RetryRequestedAt` (`DateTimeOffset?`). This is the only schema change in v0.3. It provides an audit trail distinguishing automatic retries from manually triggered ones.

No other domain models are changed.

### `IWebhookOutboxRepository` Extensions

Two new methods added to the existing interface:

```csharp
Task<(IReadOnlyList<WebhookOutboxEntry> Items, int Total)> ListByWebhookAsync(
    Guid webhookId, int page, int pageSize, CancellationToken ct = default);

Task<WebhookOutboxEntry?> GetByIdAsync(
    Guid deliveryId, CancellationToken ct = default);
```

`GetByIdAsync` must enforce tenant isolation: join through `Webhooks` to verify `TenantId` matches the caller's tenant before returning the entry.

### New CQRS

- `ListWebhookDeliveriesQuery` / `ListWebhookDeliveriesQueryHandler`
- `RetryWebhookDeliveryCommand` / `RetryWebhookDeliveryCommandHandler`
- `RetryWebhookDeliveryCommandValidator` — verifies `Status == Failed`; returns `ValidationException` (→ 422) otherwise

## Pillar 2 — Per-Tenant Rate Limiting

### Approach

Use `Microsoft.AspNetCore.RateLimiting` (built into .NET 7+, no new NuGet dependency). Sliding window policy partitioned by `tenant_id` claim value.

### Configuration (`appsettings.json`)

```json
"RateLimiting": {
  "DefaultWindowSeconds": 60,
  "DefaultPermitLimit": 300,
  "QueueLimit": 0,
  "TenantOverrides": {
    "tenant-uuid-here": { "PermitLimit": 1000 }
  }
}
```

`RateLimitingOptions` — strongly-typed settings class in `Chronith.API/Settings/RateLimitingOptions.cs`.

### Policy

- Policy name: `"per-tenant"`
- Partition key: `tenant_id` claim value (present on both JWT and API key `ClaimsPrincipal`)
- Algorithm: `SlidingWindowRateLimiter`
- Queue limit: 0 (reject immediately, do not queue)
- On rejection: `429 Too Many Requests` + `Retry-After` header + RFC 7807 Problem Details body (consistent with existing `ExceptionHandlingMiddleware` pattern)
- Response headers on every non-rejected request: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` (Unix epoch seconds)

### Placement

Rate limiting middleware registered in `Program.cs` after `UseAuthentication()` / `UseAuthorization()` so the `ClaimsPrincipal` is fully populated when the partition key is extracted. Unauthenticated requests that reach the limiter fall back to partition key `"anonymous"` (all endpoints require auth, so this path is only hit if auth middleware is misconfigured).

### `IRateLimitStore` Abstraction

Defined in `Chronith.Application/Interfaces/IRateLimitStore.cs`:

```csharp
public interface IRateLimitStore
{
    int GetPermitLimit(string tenantId);
}
```

`InMemoryRateLimitStore` in `Chronith.Infrastructure/` reads from `IOptions<RateLimitingOptions>` and applies per-tenant overrides. This allows a Redis-backed implementation to be swapped in later without changing `Program.cs`.

**Note:** The built-in `SlidingWindowRateLimiter` manages its own in-memory counter state. `IRateLimitStore` wraps only configuration retrieval (per-tenant limit lookup), not limiter state.

## Pillar 3 — OpenAPI / Swagger Documentation

### Package

`FastEndpoints.Swagger` — the official FastEndpoints integration. No manual route scanning required; it integrates natively with the `IEndpoint` pattern already in use.

### `Program.cs` Additions

```csharp
builder.Services.SwaggerDocument(o =>
{
    o.DocumentSettings = s =>
    {
        s.Title = "Chronith API";
        s.Version = "v0.3";
        s.Description = "Multi-tenant booking engine REST API";
    };
    o.EnableJWTBearerAuth();
});
```

API key scheme (`X-Api-Key` header) documented as a second security scheme via `AddSecurityScheme`.

Swagger UI served at `/docs` via `app.UseSwaggerGen()`. Disabled in production:

```csharp
if (!app.Environment.IsProduction())
    app.UseSwaggerGen();
```

OpenAPI spec available at `/openapi.json`.

### Endpoint Grouping

Tags set on each endpoint to match the existing folder structure: `Bookings`, `BookingTypes`, `Availability`, `Webhooks`, `Tenant`, `Payments`. All endpoints already have `Summary` strings — no additional XML comments on DTOs are required for v0.3.

### XML Documentation File

`<GenerateDocumentationFile>true</GenerateDocumentationFile>` added to `Chronith.API.csproj`. `1591` (missing XML comment) warning suppressed project-wide to avoid noise on non-annotated members.

## Pillar 4 — Tenant Usage Metrics

### New Endpoint

| Method | Path              | Role                         |
| ------ | ----------------- | ---------------------------- |
| `GET`  | `/tenant/metrics` | `TenantAdmin`, `TenantStaff` |

### Response Shape

```json
{
  "bookings": {
    "total": 1240,
    "by_status": {
      "pending_payment": 80,
      "pending_verification": 0,
      "confirmed": 980,
      "cancelled": 180
    },
    "this_month": 127
  },
  "webhooks": {
    "total_deliveries": 3200,
    "delivered": 3100,
    "failed": 100,
    "delivery_rate_pct": 96.9
  },
  "booking_types": {
    "active": 5,
    "archived": 2
  }
}
```

`this_month` — bookings created on or after the first second of the current calendar month in the tenant's IANA timezone. Uses the existing `TenantTimeZone.ConvertToUtc` method to compute the UTC boundary.

`delivery_rate_pct` = `delivered / (delivered + failed) * 100`, rounded to 1 decimal place. Returns `null` if both `delivered` and `failed` are 0.

`archived` booking types = rows where `IsDeleted = true`. Requires `IgnoreQueryFilters()` scoped to this query only (the global EF query filter excludes soft-deleted rows everywhere else). The query fetches all booking types for the tenant ignoring filters, then counts by `IsDeleted`.

### CQRS

- `GetTenantMetricsQuery` (no input parameters — tenant resolved from `ITenantContext`)
- `GetTenantMetricsQueryHandler` — three async queries issued in parallel via `Task.WhenAll`:
  1. Booking counts grouped by `Status`, plus a filtered count for `CreatedAt >= monthStartUtc`
  2. `WebhookOutboxEntries` counts grouped by `OutboxStatus` (only `Delivered` and `Failed` are surfaced)
  3. BookingType counts with `IgnoreQueryFilters()`, grouped by `IsDeleted`

All queries use `AsNoTracking`. No entity hydration — pure `GroupBy` + `Count` projections.

### Response Headers

`Cache-Control: max-age=60, private` — informs clients the response can be cached for up to 60 seconds.

## Out of Scope for v0.3

- Redis-backed rate limiter (deferred to v0.4)
- Maya / GCash payment providers
- User registration / login / token refresh
- Email / SMS notifications
- Staff and resource management
- Per-endpoint rate limit overrides via API
- XML doc comments on DTO properties (OpenAPI polish — v0.4)
- Webhook delivery log retention / pruning policy

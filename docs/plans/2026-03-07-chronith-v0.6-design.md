# Chronith v0.6 Design — Staff, Lifecycle, Notifications, Analytics & Public Booking

**Date:** 2026-03-07
**Prerequisite:** v0.5 (Payments & Pricing) must be completed and tagged before v0.6 implementation begins.
**Branch:** `feat/v0.6-staff-notifications-analytics` (from `develop`)

## Overview

v0.6 is a large milestone spanning 5 pillars. Each pillar is designed to be implemented in dependency order — later pillars build on earlier ones.

| Pillar | Theme                          | Key Deliverables                                                             |
| ------ | ------------------------------ | ---------------------------------------------------------------------------- |
| 1      | Staff & Resource Management    | StaffMember model, assignment, staff availability                            |
| 2      | Booking Lifecycle Enhancements | Reschedule, waitlist, time-off, custom fields                                |
| 3      | Notifications & Reminders      | Multi-channel (email/SMS/push), configurable per tenant, scheduled reminders |
| 4      | Analytics & Reporting          | Booking/revenue/utilization JSON API endpoints                               |
| 5      | Integrations & Public Booking  | iCal feed, public unauthenticated booking API                                |

## Pillar 1: Staff & Resource Management

### Domain Model

```
StaffMember
├── Id: Guid
├── TenantId: Guid
├── TenantUserId: Guid (FK → TenantUser, nullable for external staff)
├── Name: string
├── Email: string
├── IsActive: bool
├── IsDeleted: bool (soft delete)
├── CreatedAt: DateTimeOffset
└── AvailabilityWindows: List<StaffAvailabilityWindow>

StaffAvailabilityWindow
├── Id: Guid
├── StaffMemberId: Guid (FK)
├── DayOfWeek: int (0-6)
├── StartTime: TimeOnly
└── EndTime: TimeOnly
```

**BookingType changes:**

- Add `RequiresStaffAssignment: bool` (default false). When true, bookings must have a StaffMemberId.
- Add many-to-many `BookingTypeStaffAssignment` linking table — controls which staff can serve which booking types.

**Booking changes:**

- Add `StaffMemberId: Guid?` (nullable FK). Populated when staff is assigned.

### Availability Impact

When `RequiresStaffAssignment` is true, the availability query must intersect booking type windows with assigned staff windows. A slot is only available if at least one assigned staff member has an open window AND is not already booked at that time.

### Endpoints

| Method | Route                                | Access                             | Description                                     |
| ------ | ------------------------------------ | ---------------------------------- | ----------------------------------------------- |
| POST   | `/staff`                             | TenantAdmin                        | Create staff member                             |
| GET    | `/staff`                             | TenantAdmin, TenantStaff           | List staff members                              |
| GET    | `/staff/{id}`                        | TenantAdmin, TenantStaff           | Get staff member                                |
| PUT    | `/staff/{id}`                        | TenantAdmin                        | Update staff member (name, email, availability) |
| DELETE | `/staff/{id}`                        | TenantAdmin                        | Soft-delete staff member                        |
| POST   | `/bookings/{id}/assign-staff`        | TenantAdmin, TenantStaff           | Assign staff to booking                         |
| GET    | `/staff/{id}/availability?from=&to=` | TenantAdmin, TenantStaff, Customer | Get staff-specific availability                 |

### Constraints

- A staff member can only be assigned to one booking per time slot (no double-booking).
- Deactivating a staff member does not cancel existing bookings but prevents new assignments.
- Staff availability windows follow the same DayOfWeek + StartTime/EndTime pattern as booking type windows.

## Pillar 2: Booking Lifecycle Enhancements

### 2a. Reschedule

**New state transition:** Any non-cancelled, non-completed booking can be rescheduled.

- Validates new time against availability (including staff availability if assigned).
- Preserves booking ID, payment status, and staff assignment.
- Records a `BookingStatusChange` with `Rescheduled` event type (new enum value on a separate `BookingEvent` or reuse status change with metadata).
- Triggers `booking.rescheduled` webhook/notification events.

**Endpoint:** `POST /bookings/{id}/reschedule`
**Request body:** `{ "newStartTime": "2026-07-01T10:00:00Z" }`
**Access:** TenantAdmin, TenantStaff, Customer (own booking only)

### 2b. Waitlist

**Domain Model:**

```
WaitlistEntry
├── Id: Guid
├── TenantId: Guid
├── BookingTypeId: Guid
├── StaffMemberId: Guid? (optional preference)
├── CustomerId: string
├── CustomerEmail: string
├── DesiredStart: DateTimeOffset
├── DesiredEnd: DateTimeOffset
├── Status: WaitlistStatus (Waiting, Offered, Expired, Converted)
├── OfferedAt: DateTimeOffset?
├── ExpiresAt: DateTimeOffset? (offer expiration)
├── CreatedAt: DateTimeOffset
└── IsDeleted: bool
```

**Flow:**

1. Customer tries to book a full slot → API returns 409 with option to join waitlist.
2. `POST /booking-types/{slug}/waitlist` creates a WaitlistEntry.
3. When a cancellation opens a slot, a background service finds matching waitlist entries (FIFO).
4. The top entry gets an "offer" (status → Offered, notification sent). Offer expires after configurable TTL (default 30 minutes).
5. If customer accepts within TTL → booking created, entry status → Converted.
6. If expired → next entry in queue gets the offer.

**Endpoints:**

| Method | Route                            | Access                      | Description              |
| ------ | -------------------------------- | --------------------------- | ------------------------ |
| POST   | `/booking-types/{slug}/waitlist` | Customer                    | Join waitlist for a slot |
| GET    | `/booking-types/{slug}/waitlist` | TenantAdmin, TenantStaff    | List waitlist entries    |
| DELETE | `/waitlist/{id}`                 | Customer (own), TenantAdmin | Remove from waitlist     |
| POST   | `/waitlist/{id}/accept`          | Customer                    | Accept waitlist offer    |

### 2c. Time-Off / Blocked Dates

**Domain Model:**

```
TimeBlock
├── Id: Guid
├── TenantId: Guid
├── BookingTypeId: Guid? (null = applies to all types)
├── StaffMemberId: Guid? (null = applies to all staff)
├── Start: DateTimeOffset
├── End: DateTimeOffset
├── Reason: string?
├── IsRecurring: bool (false for one-off blocks)
├── CreatedAt: DateTimeOffset
└── IsDeleted: bool
```

**Availability Impact:** Time blocks subtract from available slots. The availability query checks for overlapping time blocks and excludes those periods.

**Endpoints:**

| Method | Route                    | Access                   | Description               |
| ------ | ------------------------ | ------------------------ | ------------------------- |
| POST   | `/time-blocks`           | TenantAdmin              | Create time block         |
| GET    | `/time-blocks?from=&to=` | TenantAdmin, TenantStaff | List time blocks in range |
| DELETE | `/time-blocks/{id}`      | TenantAdmin              | Remove time block         |

### 2d. Custom Fields

**Domain Model:**

```
BookingType changes:
├── CustomFieldSchema: string? (JSONB, nullable)
│   Schema format: [{ "name": "notes", "type": "text", "required": false },
│                    { "name": "guests", "type": "number", "required": true }]

Booking changes:
├── CustomFields: string? (JSONB, nullable)
│   Data format: { "notes": "Window seat", "guests": 3 }
```

**Validation:** When creating a booking, if the booking type has a `CustomFieldSchema`, the `CustomFields` payload is validated against it:

- Required fields must be present.
- Types are checked (text, number, boolean, date).
- Unknown fields are rejected.

Supported field types: `text`, `number`, `boolean`, `date`.

No separate endpoints — schema is managed via the existing `PUT /booking-types/{slug}` endpoint, and custom field values are passed in the `POST /booking-types/{slug}/bookings` request body.

## Pillar 3: Notifications & Reminders

### Channel Abstraction

```csharp
public interface INotificationChannel
{
    string ChannelType { get; } // "email", "sms", "push"
    Task SendAsync(NotificationMessage message, CancellationToken ct);
}

public record NotificationMessage(
    string Recipient,        // email address, phone number, or device token
    string Subject,
    string Body,
    string TemplateId,       // for future template support
    IDictionary<string, string> Metadata);
```

**Implementations:**

- `SmtpEmailChannel` — SMTP via `System.Net.Mail` or a library like MailKit. Config: SMTP host, port, credentials, from address.
- `TwilioSmsChannel` — Twilio REST API. Config: Account SID, auth token, from number.
- `FirebasePushChannel` — Firebase Cloud Messaging HTTP v1 API. Config: service account JSON, project ID.

### Per-Tenant Channel Configuration

**Domain Model:**

```
TenantNotificationConfig
├── Id: Guid
├── TenantId: Guid
├── ChannelType: string ("email" | "sms" | "push")
├── IsEnabled: bool
├── Settings: string (JSONB — channel-specific config: SMTP host, Twilio SID, etc.)
├── CreatedAt: DateTimeOffset
└── UpdatedAt: DateTimeOffset
```

Tenants configure which channels are active and provide credentials via `PUT /tenant/notifications/{channelType}`.

### Event-Driven Notifications

Extend the existing MediatR notification pipeline. When `BookingStatusChangedNotification` fires, a new handler checks tenant notification config and writes to a **notification outbox** (same pattern as webhook outbox).

**Events that trigger notifications:**

- `booking.confirmed` → customer gets confirmation
- `booking.cancelled` → customer gets cancellation notice
- `booking.rescheduled` → customer gets reschedule notice
- `waitlist.offered` → customer gets "slot available" alert

### Scheduled Reminders

**Domain Model:**

```
BookingType changes:
├── ReminderIntervals: string? (JSONB, nullable)
│   Format: [1440, 60]  // minutes before appointment (24h, 1h)
```

**Background Service:** `ReminderSchedulerService` (hosted service) runs on a configurable interval (e.g., every 5 minutes):

1. Query confirmed bookings with `Start` within the next reminder window.
2. For each booking, check if a reminder has already been sent for each interval.
3. If not, write a notification outbox entry.

**Tracking:** A `BookingReminder` table tracks which reminders have been sent:

```
BookingReminder
├── Id: Guid
├── BookingId: Guid
├── IntervalMinutes: int
├── SentAt: DateTimeOffset
```

### Notification Outbox

Reuse the existing outbox infrastructure (`WebhookOutboxEntry`) with a new `OutboxCategory`:

- Add `OutboxCategory.Notification` (value = 2).
- The `WebhookDispatcherService` (or a new `NotificationDispatcherService`) processes notification entries by resolving the appropriate `INotificationChannel` and sending.

### Endpoints

| Method | Route                                 | Access      | Description                       |
| ------ | ------------------------------------- | ----------- | --------------------------------- |
| GET    | `/tenant/notifications`               | TenantAdmin | List notification channel configs |
| PUT    | `/tenant/notifications/{channelType}` | TenantAdmin | Create/update channel config      |
| DELETE | `/tenant/notifications/{channelType}` | TenantAdmin | Disable channel                   |

## Pillar 4: Analytics & Reporting

### Endpoints

All analytics endpoints accept `from`, `to` (date range), and `groupBy` (day/week/month) query parameters.

| Method | Route                    | Access      | Description                                   |
| ------ | ------------------------ | ----------- | --------------------------------------------- |
| GET    | `/analytics/bookings`    | TenantAdmin | Booking counts by status, type, staff, period |
| GET    | `/analytics/revenue`     | TenantAdmin | Revenue totals by type, staff, period         |
| GET    | `/analytics/utilization` | TenantAdmin | Slot fill rates by type, staff                |

### Implementation Notes

- Analytics queries run directly against PostgreSQL with aggregate queries (COUNT, SUM, GROUP BY).
- No materialized views for v0.6 — queries hit live data. If performance becomes an issue, materialized views or a read replica can be added later.
- Results are cached in Redis with short TTL (5 minutes) keyed by tenant + date range + grouping.

## Pillar 5: Integrations & Public Booking

### 5a. iCal Feed

**Endpoint:** `GET /booking-types/{slug}/calendar.ics`

- No authentication required (public URL).
- Returns confirmed bookings for the booking type in iCalendar format (RFC 5545).
- Filterable by date range via `from` and `to` query params.
- Each booking becomes a VEVENT with summary, start/end, description.
- Calendar apps (Google Calendar, Outlook, Apple Calendar) can subscribe to this URL.

**Implementation:** Use a lightweight iCal serializer (e.g., `Ical.Net` NuGet package or hand-roll RFC 5545 output — it's a simple text format).

### 5b. Public Booking API

Unauthenticated endpoints under `/public/{tenantSlug}/` that allow customers to browse and book without JWT auth. Tenant is identified by slug in the URL path.

| Method | Route                                                                | Description                                    |
| ------ | -------------------------------------------------------------------- | ---------------------------------------------- |
| GET    | `/public/{tenantSlug}/booking-types`                                 | List active booking types                      |
| GET    | `/public/{tenantSlug}/booking-types/{btSlug}`                        | Get booking type details                       |
| GET    | `/public/{tenantSlug}/booking-types/{btSlug}/availability?from=&to=` | Check availability                             |
| GET    | `/public/{tenantSlug}/staff`                                         | List active staff (if RequiresStaffAssignment) |
| GET    | `/public/{tenantSlug}/staff/{id}/availability?from=&to=`             | Staff-specific availability                    |
| POST   | `/public/{tenantSlug}/booking-types/{btSlug}/bookings`               | Create booking                                 |
| POST   | `/public/{tenantSlug}/booking-types/{btSlug}/waitlist`               | Join waitlist                                  |

**Tenant resolution:** A middleware or endpoint filter resolves `{tenantSlug}` to a TenantId. If the slug doesn't match an active tenant, return 404.

**Rate limiting:** Public endpoints get stricter rate limits (per IP) to prevent abuse. Reuse existing rate limiting infrastructure with a separate policy.

**Customer identity:** Public bookings require `customerEmail` in the request. The CustomerId is derived from the email (or a hash). No customer account/auth for v0.6.

## Cross-Cutting Concerns

### Database Migration

One migration covering all new tables and columns:

- `staff_members` table + `staff_availability_windows` table
- `booking_type_staff_assignments` junction table
- `staff_member_id` column on `bookings`
- `requires_staff_assignment` column on `booking_types`
- `waitlist_entries` table
- `time_blocks` table
- `custom_field_schema` column on `booking_types` (JSONB)
- `custom_fields` column on `bookings` (JSONB)
- `tenant_notification_configs` table
- `booking_reminders` table
- `reminder_intervals` column on `booking_types` (JSONB)

### Testing Strategy

- Unit tests for all new domain models and state transitions.
- Integration tests for new repository methods and complex queries (analytics, availability with staff).
- Functional tests for all new endpoints.
- Performance benchmarks for analytics queries and availability with staff intersection.

### Dependency Order

```
Pillar 1 (Staff)
    ↓
Pillar 2 (Lifecycle) — depends on staff for reschedule validation, waitlist with staff preference, time-off per staff
    ↓
Pillar 3 (Notifications) — depends on lifecycle events (reschedule, waitlist offer)
    ↓
Pillar 4 (Analytics) — depends on staff and lifecycle data existing
    ↓
Pillar 5 (Public API) — depends on everything above being queryable
```

# Chronith v0.2 — Architecture Design Document

> Status: Approved
> Date: 2026-03-02
> Author: Principal Architect

---

## 1. Overview

Chronith v0.2 delivers three capabilities deferred or absent from v0.1:

1. **Real webhook delivery** — HTTP dispatch with HMAC-SHA256 signatures, outbox pattern, retry with exponential back-off.
2. **Payment provider abstraction + PayMongo integration** — `IPaymentProvider` interface with a concrete PayMongo implementation; extensible to Maya, GCash, and other providers.
3. **API key authentication** — `X-Api-Key` header alongside existing JWT Bearer; hashed storage, per-tenant key management endpoints.

All three pillars are additive — no breaking changes to existing v0.1 API contracts.

## 2. New DB Tables

### 2.1 WebhookOutboxEntries

Stores webhook payloads awaiting delivery. Written in the same transaction as the booking state change to guarantee at-least-once delivery.

```
WebhookOutboxEntry
  Id:             Guid            PK
  TenantId:       Guid            FK → Tenants.Id
  WebhookId:      Guid            FK → Webhooks.Id
  BookingId:      Guid            FK → Bookings.Id
  EventType:      string          e.g. "booking.confirmed", "booking.cancelled"
  Payload:        string          JSON body (serialized booking event)
  Status:         OutboxStatus    Pending | Delivered | Failed
  AttemptCount:   int             default 0
  NextRetryAt:    DateTimeOffset? null until first failure
  LastAttemptAt:  DateTimeOffset? null until first attempt
  DeliveredAt:    DateTimeOffset? null until success
  CreatedAt:      DateTimeOffset

OutboxStatus: Pending | Delivered | Failed
```

**Indexes:**

- `(Status, NextRetryAt)` — dispatcher poll query
- `(WebhookId)` — join on webhook lookup
- `(BookingId)` — trace deliveries per booking

### 2.2 TenantApiKeys

Stores API key metadata. The raw key is never persisted — only its SHA-256 hash.

```
TenantApiKey
  Id:          Guid            PK
  TenantId:    Guid            FK → Tenants.Id
  KeyHash:     string          SHA-256 of raw key (hex, 64 chars)
  Description: string          human-readable label
  Role:        string          TenantAdmin | TenantStaff | Customer | TenantPaymentService
  IsRevoked:   bool            default false
  CreatedAt:   DateTimeOffset
  LastUsedAt:  DateTimeOffset? updated asynchronously on each use
```

**Indexes:**

- `(KeyHash)` — unique; primary auth lookup path
- `(TenantId, IsRevoked)` — list endpoint filter

## 3. Pillar 1 — Real Webhook Delivery

### 3.1 Outbox Write

`WebhookOutboxHandler` replaces the existing stub `WebhookDeliveryHandler`. It handles `BookingStatusChangedNotification` (existing MediatR notification):

1. Load all non-deleted `Webhook` records for the booking's `BookingTypeId`.
2. Serialize a `BookingEventPayload` (booking snapshot + event type) to JSON.
3. Create one `WebhookOutboxEntry` per subscriber with `Status = Pending`.
4. Save all entries in the **same `SaveChangesAsync` call** as the booking status change (caller's transaction context).

### 3.2 WebhookDispatcherService

Background service (`IHostedService`) that runs a dispatch loop:

- Polls every **10 seconds**.
- Picks up entries where `Status = Pending AND (NextRetryAt IS NULL OR NextRetryAt <= utcNow)`.
- Processes in batches of 50.
- Dispatches HTTP POST using `IHttpClientFactory` (named client `WebhookDispatcher`).

**Request format:**

```
POST <webhook.Url>
Content-Type: application/json
X-Chronith-Event: booking.confirmed
X-Chronith-Delivery: <outboxEntry.Id>
X-Chronith-Signature: sha256=<hmac-sha256-hex>

{ ...booking event payload... }
```

HMAC key = `Webhook.Secret`. HMAC input = raw UTF-8 JSON body bytes.

**Success (HTTP 2xx):** mark `Delivered`, set `DeliveredAt = utcNow`.

**Failure (non-2xx, timeout, DNS error):**

- Increment `AttemptCount`.
- Set `NextRetryAt` using exponential back-off:

| Attempt | Delay |
| ------- | ----- |
| 1       | 30s   |
| 2       | 2m    |
| 3       | 10m   |
| 4       | 1h    |
| 5       | 4h    |

- After attempt 5: mark `Failed`.

### 3.3 Event Payload Schema

```json
{
  "event": "booking.confirmed",
  "bookingId": "...",
  "tenantId": "...",
  "bookingTypeSlug": "...",
  "status": "Confirmed",
  "start": "2026-03-06T09:00:00+00:00",
  "end": "2026-03-06T10:00:00+00:00",
  "customerId": "...",
  "customerEmail": "...",
  "occurredAt": "2026-03-02T12:00:00+00:00"
}
```

**Event type mapping:**

| State transition      | EventType                  |
| --------------------- | -------------------------- |
| → PendingVerification | `booking.payment_received` |
| → Confirmed           | `booking.confirmed`        |
| → Cancelled           | `booking.cancelled`        |

## 4. Pillar 2 — Payment Provider Abstraction + PayMongo

### 4.1 IPaymentProvider Interface (Application layer)

```csharp
// Chronith.Application/Interfaces/IPaymentProvider.cs

public interface IPaymentProvider
{
    string ProviderName { get; }

    Task<PaymentIntentResult> CreatePaymentIntentAsync(
        Booking booking,
        string currency,
        CancellationToken ct);

    bool ValidateWebhookSignature(string rawBody, string signatureHeader);

    PaymentEvent ParsePaymentEvent(string rawBody);
}

public record PaymentIntentResult(string ExternalId, string CheckoutUrl);
public record PaymentEvent(string ExternalId, bool IsPaid);
```

### 4.2 Provider Factory (Application layer)

`IPaymentProviderFactory` — resolves `IPaymentProvider` by name from a registered collection. DI registration in `Program.cs` reads `Payments:Provider` config value and registers the appropriate implementation.

### 4.3 PayMongoProvider (Infrastructure layer)

Location: `Chronith.Infrastructure/Payments/PayMongo/`

- `PayMongoProvider.cs` — implements `IPaymentProvider`
- `PayMongoOptions.cs` — `SecretKey`, `PublicKey`, `WebhookSecret`
- `PayMongoHttpClient.cs` — typed `HttpClient` with base URL and `Authorization: Basic <base64(secretKey:)>` header
- Creates a **Payment Link** via `POST https://api.paymongo.com/v1/links` using the booking amount and description
- Webhook signature validation: PayMongo signs payloads with a timestamp + HMAC-SHA256 scheme (standard PayMongo webhook format)

### 4.4 Stub Provider (Infrastructure layer / Tests)

`StubPaymentProvider` — returns a deterministic `ExternalId` and `CheckoutUrl`, always returns `true` for signature validation. Registered when `Payments:Provider = "Stub"` (used in tests and local dev without PayMongo credentials).

### 4.5 Configuration

```json
"Payments": {
  "Provider": "PayMongo",
  "Currency": "PHP",
  "PayMongo": {
    "SecretKey": "sk_test_...",
    "PublicKey": "pk_test_...",
    "WebhookSecret": "whsec_..."
  }
}
```

### 4.6 Updated Booking Flow (Automatic mode)

**CreateBookingCommand** — when `PaymentMode = Automatic`:

1. After inserting booking (status `PendingPayment`), call `IPaymentProvider.CreatePaymentIntentAsync`.
2. Store `ExternalId` as `Booking.PaymentReference`.
3. Return `CheckoutUrl` alongside the booking response.

**POST /webhooks/payment handler:**

1. Read raw body and signature header (`PayMongo-Signature` or provider-specific).
2. Call `IPaymentProvider.ValidateWebhookSignature` — return 400 if invalid.
3. Call `IPaymentProvider.ParsePaymentEvent` — if `IsPaid = true`, dispatch `PayBookingCommand` by `ExternalId` match on `PaymentReference`.

**Booking response (Automatic mode) — new field:**

```json
{
  "id": "...",
  "status": "PendingPayment",
  "checkoutUrl": "https://checkout.paymongo.com/..."
}
```

## 5. Pillar 3 — API Key Authentication

### 5.1 Key Format

```
cth_<base64url(32 random bytes)>
```

Example: `cth_X4kLmN9qR2vP8wZ3bJ6yH1cE5tA7sD0fI`

The `cth_` prefix makes keys identifiable in logs and compatible with secret scanning tools (can be added to `.gitignore` secret patterns).

### 5.2 Storage

Only `SHA-256(rawKey)` is stored in `TenantApiKeys.KeyHash`. The raw key is returned once in the `POST /tenant/api-keys` response body and never accessible again.

### 5.3 Authentication Handler

`ApiKeyAuthenticationHandler` (`IAuthenticationHandler`):

1. Reads `X-Api-Key` header. If absent, skips (returns `AuthenticateResult.NoResult()`).
2. Computes `SHA-256(rawKey)` → hex.
3. Queries `TenantApiKeys` by `KeyHash` (single indexed lookup).
4. If not found or `IsRevoked = true`: returns `AuthenticateResult.Fail(...)` → 401.
5. Builds `ClaimsPrincipal` with:
   - `tenant_id = key.TenantId`
   - `role = key.Role`
   - `sub = key.Id` (key ID as identity)
6. Fires a fire-and-forget background task to update `LastUsedAt` (non-blocking).

Both JWT and API key schemes are registered; the first to succeed wins. Order: JWT first, API key second.

### 5.4 New Endpoints

| Method   | Route                      | Roles         | Description                               |
| -------- | -------------------------- | ------------- | ----------------------------------------- |
| `POST`   | `/tenant/api-keys`         | `TenantAdmin` | Create key; returns raw key once          |
| `GET`    | `/tenant/api-keys`         | `TenantAdmin` | List keys (no raw key, no hash)           |
| `DELETE` | `/tenant/api-keys/{keyId}` | `TenantAdmin` | Revoke (soft delete, sets IsRevoked=true) |

**POST /tenant/api-keys — request:**

```json
{ "description": "PayMongo webhook service", "role": "TenantPaymentService" }
```

**POST /tenant/api-keys — response (raw key returned once):**

```json
{
  "id": "...",
  "key": "cth_X4kLmN9qR2vP8wZ3bJ6yH1cE5tA7sD0fI",
  "description": "PayMongo webhook service",
  "role": "TenantPaymentService",
  "createdAt": "2026-03-02T12:00:00+00:00"
}
```

**GET /tenant/api-keys — response:**

```json
[
  {
    "id": "...",
    "description": "PayMongo webhook service",
    "role": "TenantPaymentService",
    "isRevoked": false,
    "createdAt": "2026-03-02T12:00:00+00:00",
    "lastUsedAt": "2026-03-02T13:00:00+00:00"
  }
]
```

## 9. Test Strategy

### Unit tests (new)

- `WebhookOutboxHandler` — creates correct outbox entries per subscriber
- Retry back-off schedule — verifies `NextRetryAt` computation at each attempt
- `ApiKeyAuthenticationHandler` — hash lookup hit/miss/revoked scenarios
- `PayMongoProvider.ValidateWebhookSignature` — valid/invalid/tampered signature cases

### Integration tests (new)

- `WebhookOutboxEntries` — EF queries, indexes, status update
- `TenantApiKeys` — EF queries, hash lookup index, revoke

### Functional tests (new)

- `POST/GET/DELETE /tenant/api-keys` — happy path + auth matrix
- API key authentication — authenticated request succeeds, revoked key returns 401
- Payment webhook — valid signature accepted, invalid rejected with 400
- `POST /bookings` (Automatic mode) — response includes `checkoutUrl`

### Updated load tests

- Add `X-Api-Key` auth path to existing k6 helpers

---

## 10. Breaking Changes

None. All changes are additive:

- New endpoints
- New DB tables
- New auth scheme (opt-in alongside JWT)
- Existing booking responses gain `checkoutUrl` field (only present for Automatic payment mode)

---

## 11. Out of Scope (Deferred to v0.3+)

- User accounts / register / login (Option A auth — v0.1 design still uses external JWT for user identity)
- Maya / GCash provider implementations (interface is ready, implementations deferred)
- Delivery log endpoint (`GET /webhooks/{id}/deliveries`)
- Token refresh flow
- Webhook retry manual re-trigger endpoint

**Date:** 2026-03-07

## Overview

v0.9 ships the TypeScript SDK and a functional admin dashboard covering core booking and staff management flows. The dashboard is a separate Next.js SPA that consumes the Chronith API via the generated SDK.

| Pillar | Theme                 | Key Deliverables                                                       |
| ------ | --------------------- | ---------------------------------------------------------------------- |
| 1      | TypeScript SDK        | Auto-generated typed client from OpenAPI spec, npm package             |
| 2      | Dashboard Scaffolding | Next.js + shadcn/ui, auth flow, layout, navigation, role-based routing |
| 3      | Core Dashboard Pages  | Booking management, staff management, booking types, basic settings    |
| 4      | Data Export           | CSV/PDF export API endpoints + dashboard UI integration                |

## Pillar 1: TypeScript SDK Generation

### Approach

Auto-generate a TypeScript/JavaScript client from the Chronith OpenAPI spec (produced by FastEndpoints' Swagger integration).

### Toolchain

- **Generator:** `@hey-api/openapi-ts` — generates typed client with request/response models.
- **HTTP Client:** `fetch`-based (no axios dependency) with configurable interceptors.
- **Output:** npm package `@chronith/sdk`.

### SDK Features

- **Typed request/response models** for all endpoints.
- **Auth helpers:** `setToken(jwt)`, `setApiKey(key)`, automatic `Authorization` header injection.
- **Token refresh:** configurable callback for automatic JWT refresh on 401.
- **Error handling:** All API errors are parsed into typed `ChronithApiError` objects with `status`, `title`, `detail`, `errors` (validation), matching RFC 7807.
- **Idempotency:** `options.idempotencyKey` parameter on mutating methods.
- **Correlation ID:** Automatic `X-Correlation-Id` header on all requests.
- **Pagination helpers:** `listBookings()` returns typed paginated response with `.hasMore`, `.nextPage()`.

### CI Integration

- GitHub Actions job: on API changes, regenerate SDK, run SDK tests, bump version.
- SDK version tracks API version: `1.0.x` for `/v1/` endpoints.
- Publish to npm registry (or GitHub Packages for private use).

### Repository Location

Monorepo: `packages/sdk-typescript/` within the Chronith repository. Keeps SDK and API changes tightly coupled during development.

### Auth Flow

1. User navigates to dashboard → middleware checks for auth cookie.
2. If no cookie → redirect to `/login`.
3. Login page calls `POST /v1/auth/login` via a Next.js API route (server-side proxy).
4. API route sets JWT in an `httpOnly`, `Secure`, `SameSite=Strict` cookie.
5. Subsequent requests include cookie → Next.js API route reads cookie, adds `Authorization: Bearer {jwt}` to API calls.
6. Token refresh: TanStack Query interceptor detects 401, calls refresh endpoint, retries original request.
7. Logout: clears cookie, redirects to `/login`.

### Role-Based Routing

- `TenantAdmin` — access to all pages.
- `TenantStaff` — access to bookings (view, manage assigned), staff (view own), booking types (view). No settings, no analytics, no API keys.
- A `withAuth(allowedRoles)` HOC or middleware that checks the JWT role claim and redirects unauthorized users to a 403 page.

## Pillar 3: Core Dashboard Pages

### Booking Management

**List Page (`/bookings`):**

- Data table with columns: Customer (name + email), Type (name), Date/Time, Status (colored badge), Staff (name or "Unassigned"), Actions (view, quick-confirm, quick-cancel).
- Filters: status (multi-select), booking type (dropdown), staff member (dropdown), date range (picker), search (customer name/email).
- Pagination: 25 per page default, configurable.
- Sort: by date (default desc), by status, by customer name.
- Export button: CSV/PDF (calls export API from Pillar 4).
- "Create Booking" button (top right).

**Detail Page (`/bookings/[id]`):**

- Header: booking ID, status badge, booking type name.
- Info cards: customer info, date/time, staff assignment, payment status, custom fields.
- Status timeline: visual timeline of `BookingStatusChange` entries (date, status, changed by).
- Actions section: Confirm, Cancel (with reason dialog), Reschedule (opens modal), Assign Staff (opens modal).
- If recurring: link to recurrence rule, "Part of recurring series" badge.

**Create Booking Page (`/bookings/new`):**

- Step 1: Select booking type (card grid).
- Step 2: Select date/time (availability calendar — shows available slots fetched from API).
- Step 3: Select staff (if required — shows available staff for selected slot).
- Step 4: Customer info (select existing customer or enter new — name, email, phone).
- Step 5: Custom fields (rendered dynamically from booking type schema).
- Step 6: Review and confirm.

**Reschedule Dialog:**

- Current date/time displayed.
- New date/time picker with availability overlay (same component as create booking step 2).
- Confirm button.

**Assign Staff Dialog:**

- List of eligible staff (assigned to booking type, active, available at booking time).
- Radio select.
- Confirm button.

### Staff Management

**List Page (`/staff`):**

- Data table: Name, Email, Active (toggle badge), Assigned Types (count), Actions (view, deactivate).
- Filters: active/inactive toggle.
- "Add Staff" button.

**Detail/Edit Page (`/staff/[id]`):**

- Editable form: name, email.
- Availability editor: visual weekly calendar grid. Each day shows time blocks. Click to add/edit/delete windows. Drag to resize. Shows existing windows as colored blocks.
- Booking type assignments: multi-select checkboxes of all booking types with `RequiresStaffAssignment`.
- Save / Cancel buttons.
- Upcoming bookings section: list of next 10 bookings assigned to this staff member.

**Create Staff Page (`/staff/new`):**

- Same form as edit, without the upcoming bookings section.

### Booking Type Management

**List Page (`/booking-types`):**

- Data table: Name, Slug, Kind (Fixed/Calendar badge), Staff Required (yes/no), Custom Fields (count), Actions (view, delete).
- "Create Booking Type" button.

**Detail/Edit Page (`/booking-types/[slug]`):**

- Editable form: name, slug, description, kind, duration (Fixed only), buffer before/after (Fixed only), `RequiresStaffAssignment` toggle.
- Availability window editor: same visual component as staff availability but for the booking type's windows.
- Custom field schema editor: add/remove/reorder fields. Each field: name (text), type (dropdown: text/number/boolean/date), required (toggle). Drag to reorder.
- Payment settings: payment mode (dropdown), price.
- Reminder intervals: tag input (e.g., "24h", "1h") that maps to minutes array.
- Webhook management: list of webhooks for this type, add/delete.
- Staff assignments: which staff are assigned to this type (if `RequiresStaffAssignment`).

### Basic Settings

**General Settings (`/settings`):**

- Tenant profile form: name, slug (read-only), timezone (searchable dropdown).

**Auth Configuration (`/settings/auth`):**

- Toggle: built-in auth enabled/disabled.
- OIDC configuration: issuer URL, client ID, audience.
- Magic link toggle.
- Test OIDC connection button.

**API Key Management (`/settings/api-keys`):**

- List: name/label, created date, last used, role, expiration, actions (revoke).
- Create: name, role (dropdown), expiration (optional date picker).
- On create: modal shows the raw API key once (with copy button + warning that it won't be shown again).

## Pillar 4: Data Export

### API Endpoints

| Method | Route                                                         | Access      | Description            |
| ------ | ------------------------------------------------------------- | ----------- | ---------------------- |
| GET    | `/v1/bookings/export?format=csv&from=&to=`                    | TenantAdmin | Export bookings as CSV |
| GET    | `/v1/bookings/export?format=pdf&from=&to=`                    | TenantAdmin | Export bookings as PDF |
| GET    | `/v1/analytics/bookings/export?format=csv&from=&to=&groupBy=` | TenantAdmin | Export analytics CSV   |
| GET    | `/v1/audit/export?format=csv&from=&to=`                       | TenantAdmin | Export audit log CSV   |

### Implementation

**CSV Generation:**

- NuGet: `CsvHelper`.
- Stream directly to response body (`IAsyncEnumerable` → CSV writer → response stream) for memory-efficient handling of large datasets.
- Headers: all DTO fields as columns. Nested objects flattened (e.g., `CustomerName`, `CustomerEmail`).
- Content-Type: `text/csv`. Content-Disposition: `attachment; filename="bookings-{from}-{to}.csv"`.

**PDF Generation:**

- NuGet: `QuestPDF`.
- Generates a branded report: header (tenant name, date range, generated date), summary statistics, data table.
- Content-Type: `application/pdf`. Content-Disposition: `attachment; filename="bookings-{from}-{to}.pdf"`.

**Filters:** Same filters as the list endpoints (status, booking type, staff, date range). The export endpoints accept the same query parameters.

### Dashboard Integration

- Export button component (`ExportButton`) on all list pages.
- Dropdown: CSV or PDF.
- Triggers download via the SDK's export methods (returns blob → `URL.createObjectURL` → download link click).

### CI/CD — Dashboard

GitHub Actions workflow `dashboard-ci.yml`:

| Job               | Trigger | Description                                        |
| ----------------- | ------- | -------------------------------------------------- |
| Lint + Type Check | all     | ESLint + `tsc --noEmit`                            |
| Unit Tests        | all     | Vitest (component + hook tests)                    |
| Build             | all     | `next build` — verify production build succeeds    |
| Playwright E2E    | all     | Run against local API + dashboard (Docker Compose) |
| Docker Build      | all     | Build dashboard Docker image                       |

### Testing Strategy

**API Tests:**

- Unit + functional tests for export endpoints: CSV content validation (parse and verify rows), PDF generation (verify file headers, page count).
- SDK integration tests: instantiate generated client, call each endpoint group against a running API, verify typed responses.

**Dashboard — Component Tests (Vitest + RTL):**

- Data table: rendering, sorting, filtering, pagination, empty state.
- Forms: validation (required fields, format), submission, error display.
- Dialogs: open/close, confirm/cancel actions.
- Status badges: correct colors and labels for each booking status.
- Availability editor: add/remove/resize windows.
- Custom field editor: add/remove/reorder fields.

**Dashboard — E2E Tests (Playwright):**

| Test File               | Flows Covered                                                                                                                                           |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `auth.spec.ts`          | Login with valid credentials, login with invalid credentials, logout, unauthorized redirect, token refresh                                              |
| `bookings.spec.ts`      | List bookings, filter by status/type/date, create booking (full wizard), view detail, confirm booking, cancel booking (with reason), reschedule booking |
| `staff.spec.ts`         | List staff, create staff member, edit availability (add/remove window), assign staff to booking type                                                    |
| `booking-types.spec.ts` | List types, edit booking type, add custom field, configure availability windows                                                                         |
| `settings.spec.ts`      | Update tenant profile, create API key (verify key shown once), revoke API key                                                                           |

**Viewport Coverage:** Each E2E test runs at desktop (1280x720) and mobile (375x812) viewports.

**Test Data Seeding:** Playwright fixtures use the TypeScript SDK to seed test data (tenants, booking types, staff, bookings) before each test suite. Cleanup after.

### Deployment

**Dashboard:** Separate SPA deployed to static hosting (Vercel, Cloudflare Pages, or S3 + CloudFront). The `NEXT_PUBLIC_API_URL` environment variable points to the Chronith API.

**Docker:** Dashboard has its own `Dockerfile` (multi-stage: build with Node.js, serve with nginx or standalone Next.js). Docker Compose includes the dashboard service for local development.

    ↓

    ↓

## Pillar 1: Audit Logging

Full mutation audit trail â€” who changed what, when, with old and new values.

### Domain Model

```
AuditEntry
â”śâ”€â”€ Id: Guid
â”śâ”€â”€ TenantId: Guid
â”śâ”€â”€ UserId: string (JWT sub or API key ID)
â”śâ”€â”€ UserRole: string
â”śâ”€â”€ EntityType: string ("Booking", "BookingType", "StaffMember", etc.)
â”śâ”€â”€ EntityId: Guid
â”śâ”€â”€ Action: string ("Create", "Update", "Delete", "StatusChange")
â”śâ”€â”€ OldValues: string? (JSONB â€” serialized prior state)
â”śâ”€â”€ NewValues: string? (JSONB â€” serialized new state)
â”śâ”€â”€ Metadata: string? (JSONB â€” extra context: IP address, user agent, correlation ID)
â”śâ”€â”€ Timestamp: DateTimeOffset
```

### Implementation

**MediatR Pipeline Behavior:** `AuditBehavior<TRequest, TResponse>` intercepts all commands that implement a marker interface `IAuditable`:

1. Before handler execution: capture entity snapshot (old values) via a resolver.
2. After handler execution: capture new entity snapshot (new values).
3. Write `AuditEntry` to the database.
4. This is transparent to command handlers â€” no handler modifications needed.

**Entity Snapshot Resolution:** An `IAuditSnapshotResolver` interface with implementations per entity type. Each resolver knows how to load the entity by ID and serialize it to JSON. Registration via DI with a keyed/dictionary approach.

### Retention

- Configurable per tenant via `TenantSettings.AuditRetentionDays` (default 90 days).
- A background service (`AuditRetentionService`) runs daily and purges expired entries.

### Endpoints

| Method | Route                                                   | Access      | Description                   |
| ------ | ------------------------------------------------------- | ----------- | ----------------------------- |
| GET    | `/v1/audit?entity=&entityId=&from=&to=&userId=&action=` | TenantAdmin | Query audit log with filters  |
| GET    | `/v1/audit/{id}`                                        | TenantAdmin | Get single audit entry detail |

**Query features:** Pagination, date range filtering, entity type filtering, user filtering, full-text search on metadata.

### Database

- Table: `audit_entries` in schema `chronith`.
- Indexes: `(tenant_id, timestamp)`, `(tenant_id, entity_type, entity_id)`, `(tenant_id, user_id)`.
- No global query filter (audit entries are not soft-deleted â€” they are purged by retention).
- Partition by month if volume is high (deferred optimization).

## Pillar 2: Observability (OpenTelemetry)

### Tracing

**NuGet Packages:**

- `OpenTelemetry.Extensions.Hosting`
- `OpenTelemetry.Instrumentation.AspNetCore`
- `OpenTelemetry.Instrumentation.Http`
- `OpenTelemetry.Instrumentation.EntityFrameworkCore`
- `OpenTelemetry.Exporter.OpenTelemetryProtocol`

**Custom Activity Spans:**

- `chronith.booking.state_transition` â€” covers `Pay`, `Confirm`, `Cancel`, `Reschedule` operations.
- `chronith.payment.process` â€” covers payment provider calls.
- `chronith.webhook.dispatch` â€” covers individual webhook delivery attempts.
- `chronith.notification.dispatch` â€” covers notification channel sends.
- `chronith.availability.compute` â€” covers availability computation (includes staff intersection, time block subtraction).
- `chronith.recurring.generate` â€” covers recurring booking generation cycle.

**Span Attributes:** `tenant.id`, `booking.id`, `booking_type.slug`, `staff.id`, `user.id`, `user.role`.

**Configuration:**

```

Observability:ServiceName = chronith-api
Observability:EnableTracing = true
Observability:EnableMetrics = true
```

### Metrics

**Custom Meters:**

| Meter Name                       | Type      | Tags                        | Description                    |
| -------------------------------- | --------- | --------------------------- | ------------------------------ |
| `chronith.bookings.created`      | Counter   | `tenant_id`, `booking_kind` | Total bookings created         |
| `chronith.bookings.confirmed`    | Counter   | `tenant_id`, `booking_kind` | Total bookings confirmed       |
| `chronith.bookings.cancelled`    | Counter   | `tenant_id`, `booking_kind` | Total bookings cancelled       |
| `chronith.payments.processed`    | Counter   | `tenant_id`, `provider`     | Total payments processed       |

| `chronith.availability.duration` | Histogram | `tenant_id`, `booking_kind` | Availability query duration ms |

Standard ASP.NET Core metrics (request duration, error rate, active connections) are included automatically via the instrumentation packages.

### Structured Logging Enrichment

**Serilog Enrichers:**

- `TenantIdEnricher` â€” adds `TenantId` to all log entries from the current `ITenantContext`.
- `UserIdEnricher` â€” adds `UserId` and `UserRole` from the current JWT claims.
- `CorrelationIdEnricher` â€” adds `CorrelationId` from the `X-Correlation-Id` request header. If absent, generates a new GUID and sets it on the response header.

**Middleware:** `CorrelationIdMiddleware` â€” reads or generates `X-Correlation-Id`, stores in `HttpContext.Items`, sets on response.

### Health Checks Enhancement

- **Redis health check** â€” verifies Redis connectivity (`PING` command).
- **Background service health checks** â€” each background service (webhook dispatcher, notification dispatcher, reminder scheduler, waitlist promoter, recurring booking generator) reports healthy if its last successful run was within `2 * interval`. Tracked via a shared `IBackgroundServiceHealthTracker`.

## Pillar 3: Security Hardening

### CORS Configuration

```
Security:Cors:AllowedOrigins = ["https://dashboard.chronith.com", "http://localhost:3000"]
Security:Cors:AllowCredentials = true
Security:Cors:AllowedHeaders = ["Authorization", "Content-Type", "X-Api-Key", "Idempotency-Key", "X-Correlation-Id"]
Security:Cors:ExposedHeaders = ["X-Correlation-Id"]
```

- Configurable per environment via appsettings or environment variables.
- Supports wildcard subdomains for tenant custom domains (e.g., `*.chronith.com`).
- Applied globally via ASP.NET Core CORS middleware.

### Content Security Policy

CSP headers on all responses:

```
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' {api-origin}; frame-ancestors 'none'
```

Configurable via `Security:Csp` options. Relevant for the dashboard in v0.9.

### Rate Limiting Tuning

**Per-endpoint rate limit policies:**

| Policy          | Window    | Limit        | Key       | Applied To                      |
| --------------- | --------- | ------------ | --------- | ------------------------------- |
| `Authenticated` | 1 minute  | 300 requests | tenant_id | All authenticated endpoints     |
| `Public`        | 1 minute  | 60 requests  | IP        | `/v1/public/` endpoints         |
| `Auth`          | 5 minutes | 10 requests  | IP        | Login, register, password reset |
| `Export`        | 1 minute  | 5 requests   | tenant_id | Data export endpoints           |

**Per-tenant overrides:** Stored in `TenantSettings`. Tenants on higher plans can have relaxed limits.

### Input Validation Audit

- Review all `FluentValidation` validators for completeness.
- Add `MaximumLength` constraints on all string inputs (names: 200, emails: 320, URLs: 2048, reasons: 1000, custom field values: 5000).
- HTML sanitization: strip HTML tags from user-provided strings using a sanitizer (e.g., `HtmlSanitizer` NuGet). Applied to: customer names, staff names, booking reasons, time block reasons, custom field text values.
- JSON depth limit on JSONB inputs (custom fields, notification settings): max depth 3, max size 64KB.

### Secrets Management

**JWT Signing Key Rotation:**

- Support multiple signing keys: `Jwt:SigningKeys` array in config.
- Token validation accepts any key in the array.
- Token generation uses the first (newest) key.
- Allows rolling key rotation with zero downtime.

**Tenant Notification Credentials:**

- SMTP passwords, Twilio auth tokens, Firebase service account keys stored in `TenantNotificationConfig.Settings` are encrypted at rest.
- AES-256-GCM encryption with a master key from configuration (`Security:EncryptionKey`).
- Decrypted only when the notification dispatcher needs to send.

**API Key Rotation:**

- Document recommended rotation process in API reference.
- Add `ExpiresAt` field to `TenantApiKey` (nullable, no expiration by default).
- Background service warns (via audit log) when keys are older than configurable threshold (default 90 days).

## Pillar 4: Database Optimization

### Index Analysis

Run `EXPLAIN ANALYZE` on all critical query patterns and add composite indexes:

| Index                      | Table              | Columns                                               | Justification               |
| -------------------------- | ------------------ | ----------------------------------------------------- | --------------------------- |
| `ix_bookings_availability` | `bookings`         | `(tenant_id, booking_type_id, start, status)`         | Availability queries        |
| `ix_bookings_customer`     | `bookings`         | `(tenant_id, customer_id, created_at DESC)`           | Customer booking history    |
| `ix_bookings_staff`        | `bookings`         | `(tenant_id, staff_member_id, start)`                 | Staff schedule queries      |
| `ix_bookings_recurrence`   | `bookings`         | `(recurrence_rule_id, start)`                         | Recurring occurrence lookup |
| `ix_audit_entries_query`   | `audit_entries`    | `(tenant_id, entity_type, entity_id, timestamp DESC)` | Audit log queries           |
| `ix_audit_entries_user`    | `audit_entries`    | `(tenant_id, user_id, timestamp DESC)`                | Audit by user queries       |
| `ix_waitlist_fifo`         | `waitlist_entries` | `(tenant_id, booking_type_id, status, created_at)`    | FIFO waitlist processing    |
| `ix_idempotency_lookup`    | `idempotency_keys` | `(tenant_id, key, endpoint_route)`                    | Idempotency key lookup      |
| `ix_customers_email`       | `customers`        | `(tenant_id, email)` UNIQUE                           | Customer email uniqueness   |

### Connection Pooling

- Configure Npgsql connection pool size per environment:
  - Development: `MinPoolSize=1; MaxPoolSize=10`
  - Production: `MinPoolSize=5; MaxPoolSize=50`
- Add PgBouncer to `docker-compose.yml` for production-like testing.
- Configure `Npgsql:EnableMultiplexing=true` for better connection reuse under load.

### Materialized Views (Conditional)

If analytics queries show degraded performance under load (>500ms p95):

- Create materialized views for booking count aggregates and revenue aggregates.
- Refresh via a background service on a configurable interval (default 15 minutes).
- Analytics query handlers check for materialized view availability and fall back to live queries.

This is an optimization deferred until load testing reveals a need.

### Query Optimization

- Review all repository methods for N+1 query patterns. Add `.Include()` / `.ThenInclude()` where needed.
- Add `.AsNoTracking()` to all read-only queries (queries that don't need change tracking).
- Use `AsSplitQuery()` for queries with multiple collection includes to avoid Cartesian explosion.
- Add query tags (`TagWith()`) for easier identification in PostgreSQL query logs and `pg_stat_statements`.

### Domain Model

```
NotificationTemplate

â”śâ”€â”€ TenantId: Guid
â”śâ”€â”€ EventType: string ("booking.confirmed", "booking.cancelled", "booking.rescheduled",
â”‚                       "waitlist.offered", "reminder", "customer.welcome")
â”śâ”€â”€ ChannelType: string ("email", "sms", "push")
â”śâ”€â”€ Subject: string? (email only â€” supports variables)
â”śâ”€â”€ Body: string (supports variables)
â”śâ”€â”€ IsActive: bool
â”śâ”€â”€ CreatedAt: DateTimeOffset
â””â”€â”€ UpdatedAt: DateTimeOffset
```

### Variable Substitution

**Template Syntax:** `{{variable_name}}`

**Available Variables:**

| Variable               | Description                | Available In                     |
| ---------------------- | -------------------------- | -------------------------------- |
| `{{customer_name}}`    | Customer's display name    | All events                       |
| `{{customer_email}}`   | Customer's email address   | All events                       |
| `{{booking_date}}`     | Formatted booking date     | All booking events               |
| `{{booking_time}}`     | Formatted booking time     | All booking events (Fixed only)  |
| `{{booking_type}}`     | Booking type name          | All booking events               |
| `{{staff_name}}`       | Assigned staff member name | All booking events (if assigned) |
| `{{tenant_name}}`      | Tenant business name       | All events                       |
| `{{booking_id}}`       | Booking ID (short format)  | All booking events               |
| `{{cancel_reason}}`    | Cancellation reason        | booking.cancelled                |
| `{{new_date}}`         | New date after reschedule  | booking.rescheduled              |
| `{{new_time}}`         | New time after reschedule  | booking.rescheduled              |
| `{{waitlist_expires}}` | Offer expiration time      | waitlist.offered                 |
| `{{reminder_time}}`    | Time until appointment     | reminder                         |

**Engine:** Simple string replacement â€” no need for a full template engine. A `TemplateRenderer` service iterates over variables and replaces `{{key}}` with values from a `Dictionary<string, string>` context.

### Default Templates

When a tenant is created, default templates are seeded for all event type + channel type combinations. Examples:

**Email â€” booking.confirmed:**

- Subject: `Booking Confirmed â€” {{booking_type}}`
- Body: `Hi {{customer_name}},\n\nYour booking for {{booking_type}} on {{booking_date}} at {{booking_time}} has been confirmed.\n\nBooking ID: {{booking_id}}\n\nThank you,\n{{tenant_name}}`

**SMS â€” reminder:**

- Body: `Reminder: Your {{booking_type}} appointment is in {{reminder_time}}. â€” {{tenant_name}}`

### Endpoints

| Method | Route                                                 | Access      | Description              |
| ------ | ----------------------------------------------------- | ----------- | ------------------------ |
| GET    | `/v1/tenant/notification-templates`                   | TenantAdmin | List all templates       |
| GET    | `/v1/tenant/notification-templates/{id}`              | TenantAdmin | Get template             |
| PUT    | `/v1/tenant/notification-templates/{id}`              | TenantAdmin | Update template          |
| POST   | `/v1/tenant/notification-templates/{id}/preview`      | TenantAdmin | Preview with sample data |
| POST   | `/v1/tenant/notification-templates/reset/{eventType}` | TenantAdmin | Reset to default         |

**Preview endpoint:** Accepts optional `sampleData` in the request body (key-value pairs). If not provided, uses sensible defaults. Returns the rendered subject + body.

### Integration with Notification Dispatcher

The `NotificationDispatcherService` (from v0.6) is modified to:

1. Load the tenant's template for the event type + channel type.
2. Build the variable context from the booking/customer/staff data.
3. Render the template via `TemplateRenderer`.
4. Send via the appropriate `INotificationChannel`.

If no template exists (e.g., tenant deleted it), fall back to a built-in default.

## Cross-Cutting Concerns

### Database Migration

One migration covering:

- `audit_entries` table with indexes
- `notification_templates` table
- `expires_at` column on `tenant_api_keys` (nullable)
- New composite indexes on existing tables (bookings, waitlist, etc.)
- Seed default notification templates (via a data migration or application-level seeding)

### Testing Strategy

- **Unit tests:** Audit behavior (before/after snapshot capture), template variable substitution (all variables, missing variables, unknown variables), CORS configuration validation, input sanitization.
- **Integration tests:** Audit persistence and querying, OTel span/metric export verification, database index effectiveness (EXPLAIN ANALYZE in tests), template CRUD.
- **Functional tests:** Audit endpoints (query with filters, get detail), template endpoints (CRUD, preview, reset), security headers verification (CORS, CSP), rate limit enforcement.
- **Test coverage audit:** Run `coverlet` across all test projects. Identify gaps in Domain + Application layers. Target >95% line coverage on those layers.

### Dependency Order

```
Pillar 1 (Audit Logging) â€” independent
Pillar 2 (Observability) â€” independent
Pillar 3 (Security Hardening) â€” independent
Pillar 4 (Database Optimization) â€” independent
Pillar 5 (Notification Templates) â€” independent (modifies notification dispatcher from v0.6)
```

All five pillars are independent and can be implemented in parallel or any order. Recommend starting with Pillar 1 (Audit) as it provides immediate operational value.

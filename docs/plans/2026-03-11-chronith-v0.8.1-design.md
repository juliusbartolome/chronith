# Chronith v0.8.1 — Gap Fixes Design

**Date:** 2026-03-11
**Branch:** `feat/v0.8.1-gap-fixes`
**Tag target:** `v0.8.1`

---

## Overview

v0.8.1 is a patch release that closes four implementation gaps discovered across v0.7 and v0.8:

| # | Gap | Severity |
|---|-----|----------|
| 1 | `RecurringBookingGeneratorService.GenerateBookingsAsync` is a TODO stub — never creates bookings | High |
| 2 | Magic link flow is unimplemented — `CustomerRegisterCommandHandler` ignores `MagicLinkEnabled` | High |
| 3 | `SanitizationBehavior` is a no-op — skips all `init`-only properties (every command record) | Medium |
| 4 | v0.8 design doc says 301 but implementation correctly uses 308 — doc needs updating | Low |

No new features. No schema changes for gaps 1, 3, 4. Gap 2 (magic link) requires no EF migration because `Customer` already has the fields needed (`AuthProvider`, `IsEmailVerified`, `PasswordHash` nullable).

## Gap 1: RecurringBookingGeneratorService

### Current State

`GenerateBookingsAsync` calls `rule.ComputeOccurrences(today, horizon)` and logs the results but has an explicit `// TODO: Create bookings from computed occurrences` comment. No `Booking` records are ever written.

### Design

For each active `RecurrenceRule` and each computed occurrence date:

1. Fetch `BookingType` cross-tenant via `IBookingTypeRepository.GetByIdAcrossTenantsAsync` (ignores tenant query filter, checks `!IsDeleted`).
2. Fetch `Tenant` cross-tenant via `ITenantRepository.GetByIdAsync` (already exists; used for timezone).
3. Fetch `Customer` cross-tenant via `ICustomerRepository.GetByIdAcrossTenantsAsync` (new, same pattern).
4. Construct `DateTimeOffset start` from `occurrence + rule.StartTime` in the tenant timezone. `end = start + rule.Duration`.
5. Compute effective conflict range: `bookingType.GetEffectiveRange(start, end)`.
6. Call `IBookingRepository.CountConflictsAsync` with `[PendingPayment, PendingVerification, Confirmed]` statuses. If `conflictCount >= bookingType.Capacity`, skip this occurrence (slot already filled — could be from a previous generator run or a manual booking).
7. Create booking: `Booking.Create(rule.TenantId, rule.BookingTypeId, start, end, customerId, customerEmail, amountInCentavos, currency)`.
8. `await bookingRepo.AddAsync(booking, ct)` then `await unitOfWork.SaveChangesAsync(ct)`.
9. Publish `BookingStatusChangedNotification` so the notification pipeline fires normally.
10. After all occurrences for a rule: if `rule.MaxOccurrences` is set and total bookings created (counting existing) would hit the limit, or if `rule.SeriesEnd` has passed, call `rule.SoftDelete()` to deactivate it and persist.

### New Interface Methods

```csharp
// IBookingTypeRepository
Task<BookingType?> GetByIdAcrossTenantsAsync(Guid bookingTypeId, CancellationToken ct = default);

// ICustomerRepository
Task<Customer?> GetByIdAcrossTenantsAsync(Guid customerId, CancellationToken ct = default);
```

Both implementations use `.IgnoreQueryFilters()` + explicit `!entity.IsDeleted` filter, consistent with the existing `GetAllActiveAcrossTenantsAsync` pattern in `IRecurrenceRuleRepository`.

### Resilience

- If `BookingType` or `Customer` is not found (deleted since rule was created), log a warning and skip the rule.
- Each rule's bookings are saved independently. A failure on one rule does not abort others (the outer `try/catch` in `ExecuteAsync` already handles this at the generator level; the inner loop catches per-rule exceptions).

## Gap 2: Magic Link Registration

### Current State

`TenantAuthConfig.MagicLinkEnabled` is persisted and retrievable but `CustomerRegisterCommandHandler` ignores it entirely — it always hashes a password and issues tokens.

### Design

#### New Components

**`CustomerMagicLinkRegisterCommand`** (Application layer)
- Route: `POST /public/{tenantSlug}/auth/magic-link/register`
- Request: `{ email, name, phone? }` — no password field
- Creates `Customer` with `passwordHash = null`, `authProvider = "magic-link"`, `IsEmailVerified = false`
- Generates a signed JWT via `ITokenService.CreateMagicLinkToken(customer, tenantSlug)`:
  - Claims: `sub` (customer ID), `email`, `tenantSlug`, `purpose = "magic-link-verify"`
  - Expiry: 24 hours
  - Same signing key as regular tokens
- Sends email via `INotificationChannel` (email channel): subject "Verify your email", body contains the verification link `{baseUrl}/public/{tenantSlug}/auth/magic-link/verify?token={jwt}`
- Returns `MagicLinkInitiatedDto(string Message)` with a 202 Accepted

**`CustomerMagicLinkVerifyCommand`** (Application layer)
- Route: `POST /public/{tenantSlug}/auth/magic-link/verify`
- Request: `{ token: string }`
- Validates JWT: checks `purpose = "magic-link-verify"`, expiry, `tenantSlug` claim matches path param
- Loads customer by `sub` claim via `ICustomerRepository.GetByIdAsync`
- Calls `customer.MarkEmailVerified()`
- Issues access token + refresh token normally
- Returns `CustomerAuthTokenDto`

**Guard in existing `CustomerRegisterCommandHandler`**
- After the `AllowBuiltInAuth` check, add: if `authConfig?.MagicLinkEnabled == true`, throw `InvalidOperationException("Magic link is enabled for this tenant. Use POST /public/{tenantSlug}/auth/magic-link/register instead.")`

#### `ITokenService` addition

```csharp
/// <summary>
/// Creates a 24-hour signed JWT for magic link email verification.
/// Claims: sub (customer ID), email, tenantSlug, purpose="magic-link-verify"
/// </summary>
string CreateMagicLinkToken(Customer customer, string tenantSlug);
```

#### New DTO

```csharp
public sealed record MagicLinkInitiatedDto(string Message);
```

#### No EF Migration Needed

`Customer` already has:
- `PasswordHash` (nullable `string?`) — set to `null` for magic link customers
- `IsEmailVerified` (`bool`) — set to `false` initially, `true` after verify
- `AuthProvider` (`string`) — set to `"magic-link"`

## Gap 3: SanitizationBehavior

### Current State

The behavior filters properties via `!IsInitOnly(p)`, which returns `true` for all `required init` properties — i.e., every string property on every command record. Net effect: nothing is ever sanitized.

### Design

For `init`-only properties, the standard `property.SetValue(request, sanitized)` call is blocked by the CLR because `IsExternalInit` is a required modifier on the setter. The fix uses the same technique as `SetBackingField` in Infrastructure entity mappers:

1. Detect that a property is `init`-only (keep existing `IsInitOnly` check).
2. For those properties, find the compiler-generated backing field: `typeof(TRequest).GetField($"<{property.Name}>k__BackingField", BindingFlags.NonPublic | BindingFlags.Instance)`.
3. If the backing field is found, call `field.SetValue(request, sanitized)` directly, bypassing the init accessor.
4. If not found (e.g., hand-written property), skip (same as before).

Regular settable properties continue to use `property.SetValue` as before.

## Gap 4: Version Redirect Status Code — Doc Fix

### Current State

`src/Chronith.API/Middleware/VersionRedirectMiddleware.cs` and `tests/Chronith.Tests.Functional/Versioning/VersionRedirectTests.cs` both use **308 Permanent Redirect**. The v0.8 design doc (`docs/plans/2026-03-07-chronith-v0.8-design.md`) incorrectly states "301 Moved Permanently".

### Design

308 is the correct choice: unlike 301, it guarantees the HTTP method (POST, PUT, etc.) is preserved on redirect. The implementation is correct. Update the design doc only — no code changes.
